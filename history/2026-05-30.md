## 아직 남은 리스크

- fixed LHP sleep-before-poll 코드는 `dpas-kernel/`에 붙었지만 아직 VM runtime에는 올려보지 않았다.
- 현재 fixed LHP는 `io_poll_delay`/`q->poll_nsec` 기반의 고정 duration sleep만 구현한 단계다. adaptive PAS, UNDER/OVER 판정, bucket update는 아직 붙이지 않았다.
- `blk_mq_poll_pas_prepare()`는 여전히 no-sleep checkpoint hook이다. PAS 본체가 아니라 `pas_enabled=1`일 때 `q->last_poll_count++`만 하는 임시 확인 지점이다.
- `q->last_poll_count++`는 임시 counter라 멀티코어/멀티스레드 통계로 쓰기에는 race가 있다.
- fixed LHP 중복 sleep guard는 5.18의 request flag가 아니라 `BIO_LHP_POLL_SLEPT` bio flag를 쓴다. 최신 경로에서는 `bio_poll()`이 `bio`를 넘겨주기 때문에 이 방식이 더 자연스럽지만, 실제 runtime에서 같은 bio에 대해 sleep이 한 번만 걸리는지 확인해야 한다.
- `io_poll_delay` sysfs store는 다시 `q->poll_nsec`를 설정하도록 복구했지만, 실제 장치에서 값 설정과 sleep duration 효과는 아직 확인 전이다.
- 5.18식 `request_queue` 직접 필드 방식은 빠른 재현에는 좋지만, 장기적으로 cleanup 비용이 있다.
- `bio_poll()` 기반 DIO/HIPRI 경로와 request 기반 `blk_rq_poll()`/NVMe uring passthrough 경로는 아직 분리되어 있다.
- QEMU emulated NVMe는 `nvme_poll()` 경로 검증에는 충분하지만, 실제 물리 NVMe latency나 queue 특성을 대표하지는 않는다.

## LHP fixed sleep-before-poll 코드 리딩

5.18 `kernel/`에서 LHP sleep은 `kernel/block/blk-mq.c`의 `blk_mq_poll_hybrid()`에 있다. `blk_mq_poll()`이 classic busy poll을 바로 호출하기 전에 `q->poll_nsec != BLK_MQ_POLL_CLASSIC`이면 hybrid sleep을 먼저 시도한다.

5.18의 fixed LHP는 `q->poll_nsec > 0`일 때 `nsecs = q->poll_nsec`를 그대로 쓰는 방식이다. `q->poll_nsec == 0`이면 block poll stats 기반 adaptive LHP이고, `q->pas_enabled`이면 PAS duration update 경로로 빠진다.

5.18은 `cookie -> request` 복원이 가능해서 `RQF_MQ_POLL_SLEPT` request flag로 같은 request에 대한 중복 sleep을 막았다. 최신 `dpas-kernel` 포팅에서는 `bio_poll()`에서 `bio`를 `blk_mq_poll_bio()`까지 넘기는 구조라, 같은 역할을 하려면 request flag를 그대로 쓰기보다 bio 기반 guard가 필요하다.

최신 `dpas-kernel`의 fixed sleep 후보 위치는 `dpas-kernel/block/blk-mq.c`의 `blk_mq_poll_bio()` 안, `blk_hctx_poll()` 호출 직전이다. `nvme_poll()`이나 driver 쪽은 건드리지 않는다.

포팅 시 주의할 점은 7.1 계열 API에서는 5.18의 `hrtimer_init_sleeper_on_stack()` 대신 `hrtimer_setup_sleeper_on_stack()`을 써야 한다는 점이다. 또한 `bio_poll()` return value가 `iocb_bio_iopoll()`로도 노출될 수 있으므로, 5.18처럼 sleep 후 곧바로 `1`을 반환하기보다 sleep 후 `blk_hctx_poll()`을 호출해 실제 completion count를 반환하는 쪽이 더 안전한 후보로 보인다.

## LHP fixed sleep-before-poll 구현 및 빌드 검증

이번 세션에서는 LHP와 PAS를 분리해서 정리했다. LHP fixed sleep은 `io_poll_delay`/`q->poll_nsec` 기반으로 poll 직전에 고정 시간 잠드는 정책이고, PAS는 그 다음 단계의 adaptive sleep/bucket/update 정책이다. 따라서 이번 구현은 PAS가 아니라 fixed LHP의 최소 포팅 단계로 본다.

`io_poll_delay`는 sysfs에서 usec 단위로 입력되고, 내부 `q->poll_nsec`에는 nsec 단위로 저장된다. 예를 들어 `io_poll_delay=10`이면 `q->poll_nsec=10000`이다. `BLK_MQ_POLL_CLASSIC`은 `-1`이고, 이 값일 때는 classic busy polling 의미라 sleep duration으로 쓰지 않는다.

`dpas-kernel/block/blk-sysfs.c`에서는 기존에 `queue_poll_delay_store()`가 아무 것도 하지 않고 `return count;`만 하던 상태를 고쳤다. 이제 poll 지원 queue에서 `io_poll_delay=-1`이면 `q->poll_nsec=-1`, 양수 값이면 `val * 1000` nsec로 저장한다. show 함수도 `q->poll_nsec`를 다시 usec 단위로 보여주도록 복구했다.

`dpas-kernel/include/linux/blk_types.h`에는 `BIO_LHP_POLL_SLEPT`를 추가했다. 5.18은 `cookie -> request` 복원 후 `RQF_MQ_POLL_SLEPT` request flag로 중복 sleep을 막았지만, 최신 `dpas-kernel`에서는 검증된 hook이 `bio_poll() -> blk_mq_poll_bio()`이고 `bio`를 직접 넘기므로 bio flag를 guard로 쓰는 쪽이 현재 구조에 맞다.

`dpas-kernel/block/blk-mq.c`에는 `blk_mq_poll_lhp_sleep()` helper를 추가했다. 이 helper는 `BLK_POLL_ONESHOT`이면 빠지고, `q->poll_nsec <= 0`이면 빠지고, 이미 `BIO_LHP_POLL_SLEPT`가 있으면 빠진다. 조건을 통과하면 bio flag를 세운 뒤 `hrtimer_sleeper`를 `TASK_UNINTERRUPTIBLE` 상태로 걸고 `io_schedule()`로 fixed duration sleep을 수행한다. 타이머가 만료되면 현재 task가 다시 runnable이 되고, 이후 `hrtimer_cancel()`, `TASK_RUNNING` 복구, on-stack timer destroy로 정리한다.

호출 위치는 `blk_mq_poll_bio()` 안에서 `blk_hctx_poll()` 직전이다. 즉 driver의 `nvme_poll()`은 건드리지 않고, block layer에서 poll backend로 내려가기 전에 한 번만 sleep을 시도한다. 그 다음 기존처럼 `blk_mq_poll_pas_prepare()` no-sleep checkpoint를 거쳐 `blk_hctx_poll()`로 내려간다.

이번 구현 중 기계적으로 고친 부분도 있다. 잘못 들어간 C++ 헤더를 제거했고, `blk-mq.c`에는 `linux/hrtimer.h`를 추가했다. 사용자가 작성 중이던 예시의 `p->poll_nsec` 오타는 `q->poll_nsec`로 고쳤고, hrtimer mode 이름도 7.1 커널 API에 맞게 `HRTIMER_MODE_REL`로 정리했다. `blk_mq_poll_bio()` prototype 줄바꿈과 `if (!...)` 스타일도 checkpatch 기준에 맞췄다.
