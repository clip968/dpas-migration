## 2026-06-05 adaptive LHP 이식 시작 상태

오늘 시작 시점의 목표는 full DPAS mode switching 전에 adaptive LHP를 7.1 `dpas-kernel`에 이식하는 것이다.

현재 nested `dpas-kernel` 변경 상태:

- `block/blk-stat.h`: `blk_stats_alloc_enable()` 선언 추가.
- `block/blk-stat.c`: `blk_stats_alloc_enable()` 구현 추가.
- `block/blk-mq.c`: `blk_mq_poll_stats_start()` / `blk_mq_poll_stats_fn()` forward declaration 추가, `blk_mq_init_allocated_queue()`에서 `q->poll_cb = blk_stat_alloc_callback(...)` 시도.

현재 문제:

- `blk_mq_init_allocated_queue()`에서 `blk_mq_poll_pas_bucket`을 `blk_stat_alloc_callback()`의 bucket 함수로 넘겼지만, 이 함수는 `bio *` 기반 helper라 `request *` 기반 callback 자리와 맞지 않는다.
- 실제 빌드에서는 `blk_mq_poll_pas_bucket`이 해당 위치보다 아래에 있어 undeclared 에러가 먼저 발생했다.
- `blk_mq_poll_stats_start()`와 `blk_mq_poll_stats_fn()`은 선언만 있고 아직 정의가 없다.
- `q->poll_cb` 할당 실패 처리와 error path cleanup이 아직 없다.
- queue free 경로에서 `q->poll_stat` / `q->poll_cb` cleanup도 아직 없다.
- `io_poll_delay=0`을 허용하는 sysfs 변경도 아직 없다.
- `blk_mq_poll_lhp_sleep()`는 아직 `q->poll_nsec <= 0`이면 바로 return하므로 adaptive LHP 경로가 없다.

검증:

- `git -C dpas-kernel diff --check` 통과.
- `make O=../build/dpas-kernel-vm block/blk-mq.o` 실패.
  - 핵심 에러: `blk_mq_poll_pas_bucket` undeclared.
  - 추가 warning: `blk_mq_poll_stats_start` declared but never defined, `blk_mq_poll_stats_fn` used but never defined.

다음 수정 방향:

1. `blk_stat_alloc_callback()`용 request 기반 bucket helper `blk_mq_poll_stats_bkt(const struct request *rq)`를 새로 추가한다.
2. `blk_mq_poll_stats_fn()`과 `blk_mq_poll_stats_start()`를 실제로 정의한다.
3. `__blk_mq_end_request_acct()`에서 `RQF_STATS` completion 시 `blk_mq_poll_stats_start(rq->q)`를 호출한 뒤 `blk_stat_add(rq, now)`를 호출하게 한다.
4. `blk_mq_poll_lhp_sleep()`에서 `io_poll_delay=0`일 때 `q->poll_stat[bucket].mean / 2` 기반 sleep을 수행하게 한다.
5. `blk_mq_init_allocated_queue()` error path와 `blk_free_queue()` release path에 `poll_cb` / `poll_stat` cleanup을 넣는다.
6. `queue_poll_delay_store()`에서 `io_poll_delay=0`을 허용한다.

## 2026-06-05 adaptive LHP 코드 의미 정리

adaptive LHP의 전체 목적은 "이전 IO들이 실제로 얼마나 걸렸는지 통계를 모으고, 다음 polling 전에 그 평균의 절반만큼 잠깐 자는 것"이다.

전체 데이터 흐름:

```text
IO 완료
  -> request completion latency를 blk_stat에 기록
  -> blk_stat timer callback이 bucket별 평균 latency를 q->poll_stat에 저장
  -> 다음 bio_poll()에서 io_poll_delay=0이면 q->poll_stat[bucket].mean / 2 만큼 sleep
  -> sleep 이후 polling
```

### 1. `blk_mq_poll_stats_bkt(const struct request *rq)`

위치: `dpas-kernel/block/blk-mq.c`

역할:

- 완료된 `request`를 어느 latency 통계 bucket에 넣을지 정한다.
- read/write 방향과 IO size를 같이 반영한다.
- adaptive LHP는 모든 IO를 하나의 평균으로 보면 안 된다. 4K read, 8K read, write는 latency 특성이 다를 수 있으므로 bucket별 평균이 필요하다.

핵심 형태:

```c
ddir = rq_data_dir(rq);
sectors = blk_rq_stats_sectors(rq);
bucket = ddir + 2 * ilog2(sectors);
```

PAS 쪽의 `blk_mq_poll_pas_bucket(const struct bio *bio)`와 개념은 같지만, `blk_stat_alloc_callback()`은 `request *` 기반 bucket callback을 요구하므로 request용 helper가 따로 필요하다.

### 2. `q->poll_cb = blk_stat_alloc_callback(...)`

위치: `dpas-kernel/block/blk-mq.c`의 `blk_mq_init_allocated_queue()`

역할:

- queue 생성 시 polling latency 통계를 모을 callback 객체를 준비한다.
- 여기서 넘기는 `blk_mq_poll_stats_fn`은 통계 window가 끝났을 때 실행될 함수다.
- `blk_mq_poll_stats_bkt`는 각 request를 어느 bucket에 넣을지 정한다.

이 코드가 없으면 나중에 `blk_stat_activate_msecs()`를 호출해도 사용할 callback이 없다.

실패 경로의 `err_poll`에서 `blk_stat_free_callback(q->poll_cb)`와 `q->poll_cb = NULL`을 넣은 이유는, queue 초기화 도중 실패했을 때 이미 만든 callback을 해제하고 dangling pointer를 남기지 않기 위해서다.

### 3. `blk_stats_alloc_enable(struct request_queue *q)`

위치: `dpas-kernel/block/blk-stat.c`

역할:

- adaptive LHP가 읽을 `q->poll_stat` 배열을 처음 한 번 만든다.
- `q->poll_stat[bucket]`에는 bucket별 `mean`, `min`, `max`, `nr_samples` 같은 latency 통계가 들어간다.
- `cmpxchg(&q->poll_stat, NULL, poll_stat)`를 쓰는 이유는 여러 CPU가 동시에 처음 활성화할 수 있기 때문이다. 누군가 먼저 설치했으면 뒤늦게 만든 배열은 버린다.

중요한 점:

- 처음으로 `q->poll_stat`를 만드는 호출에서는 `false`를 반환한다.
- 그래서 adaptive LHP 첫 호출에서는 sleep이 0일 수 있다.
- completion sample이 쌓이고 timer callback이 돌아간 뒤부터 `mean / 2` 기반 sleep이 가능해진다.

### 4. `__blk_mq_end_request_acct()`

위치: `dpas-kernel/block/blk-mq.c`

역할:

- IO completion 시점에 latency sample을 blk_stat에 넣는다.
- `blk_mq_poll_stats_start(rq->q)`로 통계 수집 timer를 켠 뒤 `blk_stat_add(rq, now)`로 이번 request의 completion latency를 기록한다.

핵심 형태:

```c
if (rq->rq_flags & RQF_STATS) {
	blk_mq_poll_stats_start(rq->q);
	blk_stat_add(rq, now);
}
```

이 코드가 없으면 `q->poll_stat[bucket].mean`이 갱신되지 않으므로 adaptive LHP가 읽을 과거 latency가 없다.

### 5. `blk_mq_poll_stats_start(struct request_queue *q)`

위치: `dpas-kernel/block/blk-mq.c`

역할:

- `q->poll_stat`가 있고 callback timer가 아직 active가 아닐 때만 100ms 통계 수집 window를 시작한다.

핵심 형태:

```c
if (!q->poll_stat || blk_stat_is_active(q->poll_cb))
	return;

blk_stat_activate_msecs(q->poll_cb, 100);
```

completion마다 timer를 무작정 다시 켜지 않고, 이미 active이면 그대로 둔다.

### 6. `blk_mq_poll_stats_fn(struct blk_stat_callback *cb)`

위치: `dpas-kernel/block/blk-mq.c`

역할:

- 100ms 통계 수집 window가 끝났을 때 호출된다.
- callback 내부에 모인 `cb->stat[bucket]` 값을 `q->poll_stat[bucket]`에 복사한다.

핵심 형태:

```c
for (bucket = 0; bucket < BLK_MQ_POLL_STATS_BKTS; bucket++) {
	if (cb->stat[bucket].nr_samples)
		q->poll_stat[bucket] = cb->stat[bucket];
}
```

adaptive LHP sleep 계산 경로는 `q->poll_stat`를 읽으므로, 이 복사 과정이 필요하다.

### 7. `blk_mq_poll_lhp_nsecs(struct request_queue *q, struct bio *bio)`

위치: `dpas-kernel/block/blk-mq.c`

역할:

- 현재 bio의 bucket을 계산하고, 그 bucket의 과거 평균 latency가 있으면 평균의 절반을 sleep duration으로 반환한다.

핵심 형태:

```c
if (!blk_poll_stats_enable(q))
	return 0;

bucket = blk_mq_poll_pas_bucket(bio);
if (bucket < 0)
	return 0;

if (q->poll_stat[bucket].nr_samples)
	return (q->poll_stat[bucket].mean + 1) / 2;
```

왜 평균의 절반인가:

- LHP는 완료 예상 시간 전체를 자는 방식이 아니다.
- 평균 latency의 중간 정도까지만 자고, 나머지는 polling으로 확인한다.
- 예를 들어 같은 bucket의 과거 평균 latency가 20us라면 adaptive LHP는 약 10us sleep 후 polling한다.

### 8. `blk_mq_poll_lhp_sleep()`

위치: `dpas-kernel/block/blk-mq.c`

역할:

- fixed LHP와 adaptive LHP를 `q->poll_nsec` 값으로 분기한다.

현재 의미:

```text
q->poll_nsec < 0:
  classic polling, sleep 없음

q->poll_nsec > 0:
  fixed LHP, 사용자가 지정한 시간만큼 sleep

q->poll_nsec == 0:
  adaptive LHP, q->poll_stat[bucket].mean / 2 만큼 sleep
```

핵심 형태:

```c
if (q->poll_nsec < 0)
	return;

if (q->poll_nsec > 0)
	nsecs = q->poll_nsec;
else
	nsecs = blk_mq_poll_lhp_nsecs(q, bio);

blk_mq_poll_sleep_nsec(bio, nsecs);
```

이전에는 `q->poll_nsec <= 0`이면 바로 return했기 때문에 `io_poll_delay=0`이 adaptive LHP로 들어갈 수 없었다.

### 9. `queue_poll_delay_store()`

위치: `dpas-kernel/block/blk-sysfs.c`

역할:

- sysfs에서 `io_poll_delay=0`을 허용한다.

현재 의미:

```text
echo -1 > io_poll_delay:
  classic polling

echo 10 > io_poll_delay:
  fixed LHP 10us

echo 0 > io_poll_delay:
  adaptive LHP
```

이전에는 `val > 0`만 허용했으므로 `0`을 쓸 수 없어 adaptive LHP를 켤 방법이 없었다.

### 10. `blk_free_queue()`

위치: `dpas-kernel/block/blk-core.c`

역할:

- queue가 사라질 때 `poll_cb`와 `poll_stat`을 정리한다.

정리 순서:

```text
blk_stat_remove_callback(q, q->poll_cb)
blk_stat_free_callback(q->poll_cb)
kfree(q->poll_stat)
blk_free_queue_stats(q->stats)
blk_mq_release(q)
```

`q->poll_cb`는 `q->stats->callbacks` list에 연결될 수 있으므로, `q->stats`를 free하기 전에 callback 제거와 free를 먼저 해야 한다.

### 현재 기대 동작

```text
pas_enabled=0, io_poll_delay=-1:
  sleep 없이 classic polling

pas_enabled=0, io_poll_delay>0:
  fixed LHP sleep

pas_enabled=0, io_poll_delay=0:
  adaptive LHP
  처음에는 통계가 없어 sleep 0일 수 있음
  completion sample이 쌓이면 bucket별 mean / 2 만큼 sleep
```

### 남은 확인

빌드와 static selftest는 통과했다. 아직 남은 것은 VM runtime에서 `pas_enabled=0`, `io_poll_delay=0` 조건으로 실제 `blk_mq_poll_lhp_nsecs()`가 호출되고, `q->poll_stat[bucket].mean / 2` 값이 sleep duration으로 들어가는지 kprobe로 확인하는 것이다.

## GFP_ATOMIC 질문

사용자가 `poll_stat = kcalloc(BLK_MQ_POLL_STATS_BKTS, sizeof(*poll_stat), GFP_ATOMIC);`에서 `GFP_ATOMIC`의 의미를 질문했다.

정리:

- `GFP_ATOMIC`은 커널 메모리 할당 플래그다.
- 이 플래그를 쓰면 allocator가 메모리를 얻기 위해 sleep/blocking 할 수 없다.
- 따라서 interrupt context, spinlock을 잡은 상태, 또는 sleep하면 안 되는 빠른 경로에서 사용한다.
- 실패 가능성은 `GFP_KERNEL`보다 높으므로 호출자는 `NULL` 반환을 처리해야 한다.

이 위치에서는 polling/stat 초기화 경로에서 "잠들면 안 되는 상황일 수 있다"는 전제를 두고 즉시 가능한 메모리만 할당하려는 의미로 이해하면 된다.
