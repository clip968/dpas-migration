## 2026-06-06 06-04 PAS guard 기록 재검증과 06-05 재-poll 유도 실패 원인 분석

전날 "06-05에서 재-poll이 안 만들어진다 → 06-04 PAS guard 기록도 의심해야 한다"는 미해결 질문을, 코드 수정이나 추가 VM 실험 없이 기존 아티팩트(원본 trace, 커널 git diff, 세션 기록)로 검증했다.

### 결론

- **06-04 PAS guard 기록은 사실이다.** 원본 trace를 다시 집계해서 확인했다.
- **06-05의 "재-poll 0회"는 guard에 대한 반증이 아니라 signal 유도 실패다.** guard 판정 sample이 없었을 뿐이다.
- 원인은 커널 변경이 아니라 **signal harness 차이** — signal이 polling worker 프로세스에 닿지 않은 것으로 판단한다.

### 근거 1: 06-04 원본 trace 재검증

- `/tmp/guard-signal.trace`(06-04 14:13 생성, 14MB)가 host에 그대로 남아 있었다.
- 직접 재집계한 결과:
  - `poll_entry` 28252 = fresh(`lhp=1`) 26906 + 재-poll(`lhp=33`, guard bit 0x20) **1346**
  - `sleep_in` 26906, `sleep_ret=1` 26906 (전부 실제 sleep)
- 06-04 history 기록과 정확히 일치한다. 재-poll 1346회에서 sleep 시도 0회 — guard 차단은 실측 사실이다.

### 근거 2: 커널 diff에 재-poll 경로 변경 없음

- 06-04 guard 테스트 커널 `g8e7968051d39` → 06-05 테스트 커널 `g4be3fefb1311` 사이 커밋은 `658e97c65`(cooldn/heatup)와 `4be3fefb1`(adaptive lhp) 둘뿐이다.
- diff 확인 결과 재-poll 메커니즘 3요소는 전부 무변경:
  - `task_sigpending()` early-return: `dpas-kernel/block/blk-mq.c:5402`
  - `BIO_LHP_POLL_SLEPT` guard: `dpas-kernel/block/blk-mq.c:5443`
  - 같은 bio로 `bio_poll()`을 재호출하는 iomap 루프: `dpas-kernel/fs/iomap/direct-io.c:881-891`
- 변경된 것은 lhp sleep nsec 계산, poll stats 수집, adaptive up/dn 조정뿐이라 재-poll 발생 여부와 무관하다. "adaptive LHP 포팅이 재-poll 경로를 죽였다"는 가설은 기각.

### 근거 3: guest는 동일, signal harness가 다름

- 06-04와 06-05의 guest는 **같은 VM**이다 (`/tmp/dpas-vm-known-hosts`와 `/tmp/dpas_known_hosts`의 host key·IP `192.168.122.57` 동일). "guest/fio가 달라졌다"는 가설도 기각.
- 06-04 세션 기록에서 당시 storm 명령 원문을 확인했다:
  - fio 시작 **0.4초 후**, `pkill -USR1 -x fio`를 3ms 간격 400회.
  - `pkill -x fio`는 frontend뿐 아니라 **fork된 job worker**(실제로 poll 루프를 도는 프로세스, trace의 `fio-2922`)까지 전부 맞춘다.
  - storm이 늦게 시작해 fio 자체 USR1 핸들러(status dump) 설치 후였기 때문에 fio가 죽지 않았다.
- 06-05의 관측 세 가지 — ① fio가 USR1로 사망(exit 138), ② LD_PRELOAD shim 필요, ③ signal 164발에 재-poll 정확히 0회 — 는 전부 "signal이 polling worker에 한 번도 닿지 않았다"로 수렴한다.
- 메커니즘상 worker가 I/O in-flight 중에 USR1을 받으면 `TIF_SIGPENDING`은 syscall 복귀 전까지 풀리지 않고, iomap 루프는 completion까지 같은 bio로 무조건 재진입하므로 **guard-bit 재-poll entry가 최소 1회는 반드시 찍힌다**. 06-04 기준 worker에 닿은 signal의 약 27%(400발 중 108 인스턴스)가 재-poll burst를 만들었다. 164발이 worker에 닿고도 0회는 통계적으로 불가능하다.
- 가장 유력한 시나리오: 06-05 harness가 `pkill -x fio` 대신 frontend PID(`$!`)에만 `kill`을 쐈다. frontend는 waitpid 중에 signal을 소화하고, worker는 아무것도 보지 못한다.
- 한계: 06-05 테스트 세션의 기록은 host에 남아 있지 않아 당시 명령 원문은 확인하지 못했다. targeting 시나리오는 행동 증거로부터의 추정이다 (아래 tracepoint 검증으로 확정 가능).

### 06-05 기록 정정

- 06-05의 "2026-06-04 기록은 현재 환경에서 재현되지 않았다 → 그 기록을 의심해야 한다"는 판단은 정정한다. 원본 trace가 존재하고 기록과 일치하므로, 의심 대상은 06-04 결과가 아니라 **06-05의 signal 유도 방식**이다.

### 헷갈리기 쉬운 점

- "signal이 N발 전송됐다(`kill` 성공)"와 "polling worker의 `TIF_SIGPENDING`이 set됐다"는 다른 사실이다. 전자는 대상 PID가 존재했다는 뜻일 뿐이다.
- fio는 frontend가 job을 fork하므로, PID 하나에 `kill`하는 것과 `pkill -x fio`는 도달 범위가 다르다.
- USR1 핸들러의 종류(fio status dump든 noop shim이든)는 커널 안 `task_sigpending()` 가시성과 무관하다. delivery는 어차피 syscall 복귀 시점에만 일어난다.

### 다음 작업

1. **06-04 harness를 그대로 재사용**: guest 안에서 `pkill -USR1 -x fio`, 3ms 간격, fio 시작 0.5초 후 storm 시작 (shim은 startup race 보험으로 병행 가능).
2. 같은 trace에 `signal:signal_generate` / `signal:signal_deliver` tracepoint를 worker PID 필터로 추가해서 "signal이 worker에 닿았는가"를 추정이 아니라 데이터로 확인.
3. 위 harness로:
   1. `pas_enabled=1` 재현 run — 06-04 수치(재-poll 발생 + guard 차단)가 재현되는지 확인.
   2. adaptive LHP run (`pas_enabled=0`, `io_poll_delay=0`) — 원래 목표인 adaptive LHP guard 판정.
4. deterministic 커널 hook(`__blk_hctx_poll()` 강제 early return)은 1~3이 실패할 때만 검토.

## 2026-06-06 재-poll guard VM 검증 완료 (adaptive LHP + PAS)

위 "다음 작업" 1~3을 VM에서 실행해 닫았다. deterministic hook(4번)은 불필요했다. 상세 수치/ASCII는 `history/2026-06-06-repoll-guard-test-results.md` 참고.

- harness: 06-04 충실 복원 — `pkill -USR1 -x fio`(worker까지 타격), 3ms, fio 시작 0.5초 후, warmup 2s + run 3s. 사전 probe에서 이 커널 fio가 SIGUSR1에 죽지 않음을 확인해 **shim은 사용 안 함**(06-05 segfault는 shim+3ms 조합이 원인).
- signal tracepoint(`signal_generate`/`signal_deliver`, `sig==10`)를 같은 trace에 부착 → "signal이 worker에 닿았는가"를 추정이 아니라 데이터로 확정.

결과 (커널 `g4be3fefb1311`):

- **PAS** (`pas_enabled=1`): 재-poll 163회 발생, 그 163회는 `sleep_nsec` 진입 0(= `pas_sleep` 입구 차단). worker pid 3747이 `signal_deliver` 136회 수신 → REACHED. 06-04 결과 현재 커널 재현.
- **adaptive LHP** (`pas_enabled=0`, `io_poll_delay=0`): 재-poll 402회 발생, 그 402회 전부 `sleep_nsec` 진입했으나 `BIO_LHP_POLL_SLEPT`로 `ret0`(sleep 건너뜀). worker pid 5149가 `signal_deliver` 141회 수신 → REACHED. **원래 목표 달성.**
- 구조 차이: PAS는 `pas_sleep` 입구(상류) 차단(`sleep_in=fresh`), LHP는 `sleep_nsec` 내부 차단(`sleep_in=poll_entry`, `ret0=re-poll`). 둘 다 중복 sleep 0.

판정/정정:

- 06-05의 "재-poll 0회"는 guard 결함이 아니라 signal이 worker에 안 닿은 것이었음이 **양방향으로 확정**(worker 타격 시 재-poll 즉시 발생).
- 사후: knob 원복(`pas_enabled=0`, `io_poll_delay=-1`, `io_poll=1`), kprobe/tracepoint 제거. dmesg의 segfault는 996s stale(06-05 shim+3ms 세션, uptime 37040s)뿐이고 **오늘 run은 새 dmesg 0줄**. 두 fio run 모두 `rc=0`.

다음:

- adaptive LHP를 기준선으로 full DPAS mode switching 설계로 갈지 사용자 결정. 성능 평가는 아직 아님(이번 수치는 storm/trace 부착 상태의 경로 검증용).

- GRUB 도구 존재: `grub-reboot`, `grub-editenv`, `update-grub`, `mkinitramfs`

핵심 판정:

- **그냥 `bzImage`만 `/boot`에 넣고 host root로 1회 부팅하는 것은 현재 상태에서 위험하다.**
- 이유: host root disk가 SATA SSD인데, VM용 config는 `CONFIG_SATA_AHCI=m`이고 matching `ahci.ko`/`libahci.ko` 모듈이 없다. initramfs 없이 부팅하면 커널이 root disk를 못 보고 panic할 가능성이 높다.
- `EXT4_FS=y`, `BLK_DEV_SD=y`, `EFI_STUB=y`, `DEVTMPFS=y`, `DEVTMPFS_MOUNT=y`, `BLK_DEV_NVME=y`는 host boot에 유리하지만, SATA AHCI가 빠져 있어 현재 host root에는 충분하지 않다.
- `root=UUID=...`는 matching initramfs가 없으면 피하고, 직접 부팅 경로에서는 `root=PARTUUID=b1db2aa1-2042-4f43-b6f9-b8d916802e8e`가 더 안전하다.

가능한 다음 선택지:

1. 보수적 경로: host boot용 별도 config/build를 만들고 `CONFIG_SATA_AHCI=y` 등 root disk 필수 드라이버를 built-in으로 바꾼 뒤, GRUB custom entry + `grub-reboot`로 1회성 부팅.
2. initramfs 경로: 현재 config에서 modules를 빌드/설치하고 matching initramfs를 만든 뒤 GRUB entry를 구성.
3. 회피 경로: host 직접 부팅 대신 현재 libvirt VM direct-kernel boot 유지.

아직 하지 않은 것:

- `/boot`, `/etc/grub.d`, `/lib/modules`, EFI NVRAM 수정 없음.
- `grub-reboot` 실행 없음.
- host 재부팅 없음.

## 2026-06-06 DPAS host one-shot boot emergency mode 원인 확인

사용자가 `sudo reboot` 후 DPAS host one-shot entry로 부팅했으나 systemd emergency mode에 진입했고, 강제 재부팅 후 기본 Ubuntu 커널로 복귀했다.

현재 복귀 상태:

- 현재 커널: `7.0.0-22-generic`
- GRUB one-shot 상태: `strings /boot/grub/grubenv`에서 `next_entry=`로 비어 있음. one-shot은 소비됨.

사용자가 확인한 직전 boot 로그 핵심:

```text
Mounting boot-efi.mount - /boot/efi...
mount: /boot/efi: wrong fs type, bad option, bad superblock on /dev/sdc1, missing codepage or helper program, or other error.
kernel: FAT-fs (sdc1): IO charset iso8859-1 not found
boot-efi.mount: Failed with result 'exit-code'.
Failed to mount boot-efi.mount - /boot/efi.
Dependency failed for local-fs.target - Local File Systems.
Started emergency.service - Emergency Shell.
Reached target emergency.target - Emergency Mode.
```

원인:

- DPAS host 커널은 `/` root까지는 올라왔지만, `/etc/fstab`의 `/boot/efi` vfat mount에서 실패했다.
- `/boot/efi`는 `iocharset=iso8859-1`를 요구한다.
- DPAS host config는 `CONFIG_FAT_FS=y`, `CONFIG_VFAT_FS=y`지만 `CONFIG_NLS_ISO8859_1=m`이다.
- 이번 부팅 방식은 initramfs/modules 설치 없이 `bzImage`만 직접 부팅했기 때문에 `nls_iso8859-1` 모듈을 로드할 수 없었다.
- 그 결과 `local-fs.target` dependency failure가 발생해 emergency mode로 들어갔다.

추가로 보인 문제:

- `autofs4`, `ppdev`, `lp`, `parport_pc`, `i2c_dev`, `msr` module load 실패가 보였다. `i2c_dev`는 config상 built-in이지만 systemd module-load 목록과 이름 처리 확인이 필요하다.
- `ufw`는 `nf_tables` protocol unsupported로 실패했다. DPAS host config에서 netfilter/nftables 관련 항목 다수가 `m`인데 modules가 없어 발생한 부수 문제로 보인다. emergency mode의 직접 원인은 `/boot/efi` mount 실패다.

재시도 전 결정 필요:

- 보수적 수정: `CONFIG_NLS_ISO8859_1=y`를 built-in으로 바꿔 host 커널을 재빌드한다. 필요하면 `CONFIG_NLS_UTF8=y`, `CONFIG_AUTOFS_FS=y`, `CONFIG_X86_MSR=y`, nftables/iptables/conntrack 관련 항목도 built-in 범위를 검토한다.
- 대안: DPAS 커널용 modules와 initramfs를 설치해 `m` 항목을 로드 가능하게 만든다.
- 회피: `/boot/efi` fstab에 `nofail`을 주는 방식은 실험 회피책이지만 host 설정 변경이므로 별도 사용자 결정 없이는 하지 않는다.

현재 판단:

- 기존 `/boot/vmlinuz-7.1.0-rc4-dpas-host-g4be3fefb1311`로 다시 one-shot boot하면 같은 emergency mode에 들어갈 가능성이 높다.
- Optane smoke test는 DPAS host 커널을 수정/재빌드하고 `/boot` entry를 갱신하기 전까지 진행하지 않는다.
