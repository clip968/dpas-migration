### 남은 리스크와 헷갈릴 수 있는 점

- 반복 측정은 아직 `numjobs=1`, 4KiB random read, ext4 file direct I/O 조건이다.
- mode 순서는 randomize하지 않았고 `INT -> CP -> LHP -> PAS` 고정이었다.
- CPU governor, IRQ affinity, background daemon은 아직 별도 고정하지 않았다.
- repeat 수 5회, runtime 27s라 경향 확인용이지 최종 논문용 통계로 보기는 이르다.
- PAS의 context switch 수가 높게 나와 CPU 절감의 질과 scheduler 영향은 추가 분석이 필요하다.
- LHP IOPS stdev 1737.39는 LHP 전체 반복이 계속 흔들린 것이 아니라 `repeat=1`만 72253.25 IOPS로 낮아서 생긴 값이다.
  - LHP `repeat=2..5`만 보면 mean 76132.31, stdev 110.19로 안정적이다.
  - 현재 가설: adaptive LHP의 `q->poll_stat` 기반 sleep 통계가 첫 측정 구간에서 아직 안정화되지 않았거나 이전 모드/직전 상태의 영향을 받았다. 다음 측정에서는 LHP warmup 또는 첫 반복 제외 규칙을 명시적으로 검증해야 한다.

## LHP stdev warmup 재측정 준비

- 사용자 요청: 기존 host Optane 조건은 유지하고 warmup 조건만 추가해 `INT`, `CP`, `LHP`, `PAS`를 재측정한다.
- `scripts/micro_4krr/host_repeat_optane.sh`에 optional warmup env를 추가했다.
  - `DPAS_WARMUP_RUNTIME=0`이면 기존 동작과 동일하다.
  - `DPAS_WARMUP_RUNTIME>0`이면 각 measured run 직전에 같은 mode/hipri/workload로 warmup fio를 실행하고 `warmup.json`에만 저장한다.
  - measured run 전에는 다시 `drop_caches`를 수행하므로 기존 측정의 cache 조건은 유지하고, queue-level polling 통계 warmup 효과만 확인하는 의도다.
- 실행하려던 명령:
  `sudo env DPAS_RUNTIME=27 DPAS_RAMP_TIME=3 DPAS_REPEATS=5 DPAS_WARMUP_RUNTIME=5 DPAS_WARMUP_RAMP_TIME=3 DPAS_MODES='INT CP LHP PAS' bash scripts/micro_4krr/host_repeat_optane.sh`
- 상태: `bash -n scripts/micro_4krr/host_repeat_optane.sh`는 통과했다. 실제 fio 실행은 `sudo`가 비밀번호 TTY 인증을 요구해서 Codex 세션에서 진행하지 못했다.

해석:

- LHP stdev는 이전 `1737.39`에서 `540.79`로 줄었다. CV도 `2.31%`에서 `0.52%`로 줄었다.
- LHP 첫 반복 penalty는 이전 `-5.10%`에서 이번 `-0.97%`로 줄었다. warmup이 첫 LHP run의 낮은 값 문제를 상당히 줄인 것으로 보인다.
- 단, 절대 IOPS는 모든 mode에서 크게 올랐다. CP도 `97.9k -> 116.0k`, INT도 `70.3k -> 81.0k`로 올랐으므로, 개선을 LHP 내부만의 효과로 해석하면 안 된다. warmup이 device/CPU/controller 상태를 전반적으로 바꿨거나 실행 시점 환경 차이가 섞였을 수 있다. 확인 필요.
- 이번 warmup 조건의 성능 순위는 `CP > PAS ≈ LHP > INT`에 가깝다. PAS는 LHP보다 IOPS가 약 1.3% 높고, LHP는 PAS보다 CPU total이 약 3.27%p 낮다.
