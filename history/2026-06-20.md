- 이미 완료된 결과는 `1T`의 `INT/CP/LHP/PAS/DPAS`, `4T`의 `INT/CP/LHP/PAS`까지 존재하고, `4T DPAS`와 이후 thread sweep은 실행되지 않았다.
- cleanup 뒤 `nvme1n1`은 마운트되어 있지 않고 `pas_enabled=0`, `pas_adaptive_enabled=0`, `switch_enabled=0`, `io_poll_delay=-1`로 돌아갔다. 단, `io_poll=1`은 남아 있어 다음 실험 전 의도한 상태인지 확인 필요하다.
- 근본 원인 확인 필요: 사용자 shell에서 `sudo dmesg -T`로 `fio` segfault, NVMe 오류, 커널 warning/oops가 같이 있었는지 확인해야 한다.
- 사용자 shell의 `sudo dmesg -T | grep ...` 결과, 19:16:00에 `Oops: divide error`가 발생했고 `CPU: 0 UID: 0 PID: 6347 Comm: fio`로 찍혔다. 즉 단순 fio 사용자 공간 오류가 아니라 fio가 밟은 커널 경로에서 0으로 나누는 오류가 난 상황이다.
- grep 출력에는 `RIP`/`Call Trace`가 빠져 있어 정확한 함수는 확인 필요하다. 후보는 PAS adaptive duration 계산의 `blk_mq_poll_pas_update_duration()` 또는 DPAS 평균 QD 계산 경로지만, 실패 모드가 `PAS`라서 `switch_enabled=0` 경로를 먼저 의심해야 한다.
- 전체 Oops trace 확인 결과 `RIP: blk_mq_poll_bio+0x1b8/0x800`, `Call Trace: bio_poll -> __iomap_dio_rw -> xfs_file_dio_read -> preadv2` 경로다.
- `objdump -d --line-numbers --disassemble=blk_mq_poll_bio build/dpas-kernel-host/vmlinux`로 매핑하면 `+0x1b8`는 `dpas-kernel/block/blk-mq.c:5625`의 `stat->dur = mul_u64_u64_div_u64(stat->dur, (u64)stat->adj, q->div);`다.
- fault 명령은 `divq 0x120(%r9)`이고 `pahole` 기준 `request_queue.div` offset이 `288 == 0x120`이므로 이 나눗셈의 분모는 `q->div`다.
- 단, x86 `divq`의 `divide error`는 분모 0뿐 아니라 quotient overflow에서도 발생한다. fault 시 `RDX=0x15b245`라 기본 `q->div=1000000`이어도 quotient overflow 조건이 될 수 있다.
- 더 정확한 현재 가설: `PAS` adaptive duration에서 `cur_case == 3` 반복 등으로 `stat->adj`/`stat->dur`가 커지고, 5.18식 `stat->dur * stat->adj / q->div` 계산을 `mul_u64_u64_div_u64()`로 옮기면서 기존 unsigned wrap 대신 x86 `#DE`가 발생했다.
