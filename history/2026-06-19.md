## host full-DPAS micro_4krr 결과 위치 재확인

- `scripts/micro_4krr/run_host_dpas_optane.sh`는 실제로 실행됐고 결과 파일도 남아 있다.
- 결과 위치:
  - `scripts/micro_4krr/fio_data/nvme1n1/RR/<jobs>T/<mode>/fio_report_<repeat>.log`
  - `scripts/micro_4krr/parsed_data/nvme1n1-RR-repeat_5.txt`
  - `scripts/micro_4krr/result_data/nvme1n1-RR-repeat_5.txt`
- 조건은 `INT,CP,LHP,PAS,DPAS`, jobs `20,16,8,4,2,1`, repeats `5`, RR workload였다.
- `fio_report_*.log`는 총 150개로, 6개 job x 5개 mode x 5회 반복과 맞다.
- 주의: `parse.py`는 jobs를 `20,16,8,4,2,1` 내림차순으로 쓰지만 `mean_std.py`를 `ascend=1`로 호출한다. 그래서 `result_data/nvme1n1-RR-repeat_5.txt`는 IOPS와 CPU 행이 `data idx` 안에서 섞여 있어 그대로 신뢰하면 안 된다.
- 현재는 원본 `parsed_data` 또는 `fio_data`에서 직접 평균/표준편차를 다시 계산하는 것이 맞다.

## full-DPAS 테스트 스크립트 구조 재확인

- 현재 full-DPAS host 러너는 `scripts/micro_4krr/run_host_dpas_optane.sh`다.
- 기본 조건은 device `nvme1n1`, modes `INT,CP,LHP,PAS,DPAS`, jobs `20,16,8,4,2,1`, repeat `1`, runtime `10s`, workload `4K random read`, `ioengine=pvsync2`, `iodepth=1`이다.
- CPU는 별도 고정하지 않는다. 스크립트가 `/proc/self/status`의 `Cpus_allowed_list`를 읽어 fio `--cpus_allowed`에 넘기므로, 단일 CPU 실험은 `taskset -c <cpu>`로 스크립트 전체를 묶어야 한다.
- Optane `nvme1n1`은 기본 `DPAS_THRESHOLD=30`으로 `switch_param3=30`이 된다. 이는 OL->INT 평균 QD 문턱을 QD 3.0으로 두는 theta=3 설정이다.
- PAS->CP 전환은 `dpas-kernel/block/blk-mq.c`의 `avg_qd == 10` 조건이다. 주석상 QD 1 수준을 의미하지만, "QD 1 이하"가 아니라 정확히 10으로 계산될 때만 CP로 간다.
- 기존 fio 로그에는 `dpas_mode`, `dpas_qd_sum`, window별 평균 QD가 직접 기록되지 않는다. 2~4 thread에서 왜 CP로 안 가는지 보려면 QD/mode 전환 로그나 sysfs/debug 출력 추가가 필요하다.

## micro_4krr parser thread 순서 수정

- `scripts/micro_4krr/parse.py`가 `mean_std.py`를 항상 `ascend=1`로 호출해 `20,16,8,4,2,1` 내림차순 결과의 `result_data`에서 IOPS/CPU 행이 섞였다.
- `JOB_LIST`가 엄격한 내림차순이면 `mean_std.py`에 `ascend=0`을 넘기고, 그 외에는 기존처럼 `ascend=1`을 넘기도록 수정했다.
- `/tmp` 복사본에서 기존 fio 로그로 검증했다. 내림차순 `20,16,8,4,2,1`과 오름차순 `1,2,4,8,16,20` 모두 `data idx = 0`은 IOPS, `data idx = 1`은 CPU로 분리됐다.

## DPAS switch stats sysfs 수정

- `dpas-kernel/block/blk-sysfs.c`의 `dpas_switch_stats` 출력이 window 진행용 임시 counter를 읽던 문제를 고쳤다.
- `queue_dpas_switch_stats_show()`는 이제 `dpas_pas_eval`, `dpas_pas_to_cp`, `dpas_ol_to_int`, `dpas_int_to_ol` 같은 누적 stats 필드를 lock 안에서 local 변수로 복사하고, `sysfs_emit()`는 lock 밖에서 호출한다.
- `switch_enabled` write 시 debug stats reset이 lock 밖에서 따로 실행되던 부분을 `queue_dpas_reset_switch_state()` 안으로 옮겨 mode/window reset과 같은 `dpas_lock` 범위에서 처리되게 했다.
- 정적 회귀 검사와 `git diff --check -- block/blk-sysfs.c`는 통과했다. macOS 기본 `make`는 GNU Make 3.81이라 커널 `bzImage` 빌드가 GNU Make 4.0 요구에서 중단됐다.

## VM 빌드테스트 중단 및 정리

- Colima Ubuntu 24.04 VM에서 `x86_64-linux-gnu-` cross build 환경을 준비하고 `make -C dpas-kernel O=../build/dpas-kernel-vm ARCH=x86 CROSS_COMPILE=x86_64-linux-gnu- -j4 bzImage`를 시작했다.
- 빌드 중 `block/blk-core.o`, `block/blk-sysfs.o`, `block/blk-mq.o`, `block/built-in.a`까지는 생성 로그를 확인했다. 즉 이번에 수정한 `blk-sysfs.c`와 주변 DPAS block 파일은 해당 지점까지 컴파일을 통과했다.
- 사용자가 빌드테스트는 나중에 하자고 해서 최종 `arch/x86/boot/bzImage` 생성 여부는 확인하지 않고 중단했다.
- 이후 `./vm stop`으로 VM을 종료했고, `./vm status`가 `colima is not running`을 반환해 내려간 상태를 확인했다.
- 남은 확인: VM을 다시 켠 뒤 최종 `bzImage` 빌드를 끝까지 확인하고, 부팅 후 `logging_enabled=2`, `switch_enabled` 토글, fio run 단위로 `/sys/block/<dev>/queue/dpas_switch_stats`가 기대대로 증가/reset되는지 확인한다.
