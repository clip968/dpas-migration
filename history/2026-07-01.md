# 2026-07-01 DPAS 공부/작업 현황

## 이어받은 맥락
- 전날까지의 핵심은 단일 CPU 다중 thread에서 polled direct I/O wait loop가 끝나지 않는 문제다.
- 1차 수정은 PAS sleep metadata를 stack-local이 아니라 `bio->bi_pas`에 저장해, `bio_poll()` 호출 사이에 OVER/UNDER 판정 정보를 보존하는 방향이었다.
- 이후 `blk_mq_poll_bio()`에 cookie 검증과 sleep 전 oneshot poll이 추가됐지만, 런타임에서 fio가 여전히 끝나지 않는다는 증상이 남았다.

## 오늘 확인한 코드 위치
- `dpas-kernel/fs/iomap/direct-io.c`: sync DIO wait loop는 polled bio면 `bio_poll()`을 계속 호출한다.
- `dpas-kernel/block/blk-core.c`: `bio_poll()`은 `bi_cookie == BLK_QC_T_NONE`이면 0을 반환한다.
- `dpas-kernel/block/blk-mq.c`: `__blk_hctx_poll()`은 completion을 못 찾으면 `need_resched()`까지 busy poll하고, 별도 poll-count budget은 현재 없다.
- `dpas-kernel/block/blk-mq.c`: `blk_mq_poll_bio()`는 sleep 전 oneshot poll을 추가했지만, sleep 이후 두 번째 poll은 여전히 일반 busy-poll 경로다.

## 현재 판단
- "fio runtime이 길다"기보다, 특정 sync DIO가 `bio_poll()` 반복에서 completion을 못 확인해 waiter가 풀리지 않는 경로를 먼저 의심해야 한다.
- 최근 "livelock 해결" 커밋은 invalid cookie와 이미 끝난 I/O를 sleep 전에 잡는 문제를 줄이지만, completion을 계속 못 잡는 상위 wait-loop 문제를 완전히 닫지는 않는다.
- 6월 27일 기록처럼 `bio_poll()`이 0을 반환해도 `iomap` wait loop는 곧바로 다시 `bio_poll()`을 호출하므로, 단일 CPU progress 보장은 아직 별도 검증이 필요하다.

## 다음 확인 지점
- VM 내부에서 fio task의 `wchan`, stack, `bi_cookie`/poll 경로 로그를 확인해 `bio_poll()` 반복인지, mount/umount 대기인지, NVMe timeout인지 먼저 분리한다.
- `switch_enabled`, `pas_enabled`, `pas_adaptive_enabled`, `dpas_mode`, `dpas_switch_stats`, `pas_d_init` 상태를 같은 장치에서 확인한다.
- 수정이 필요하면 poll-loop yield/budget 복원, `iomap` wait-loop 양보 지점, 또는 polled bio cookie/merge 방어 중 어느 범위를 먼저 실험할지 사용자 결정이 필요하다.

## CP 회귀 원인 후보 확인
- 루트 repo의 `dpas-kernel` 포인터는 `9193bdd03`에서 `bd3df84d9`로 이동했고, 실제 변경 파일은 `dpas-kernel/block/blk-mq.c` 하나다.
- `bd3df84d9`는 `blk_mq_poll_bio()`에서 CP/LHP/PAS 분기 전에 `__blk_hctx_poll(..., BLK_POLL_ONESHOT, ...)`를 먼저 호출한다. 따라서 LHP/PAS 방어 패치처럼 보이지만 standalone CP도 같은 새 pre-poll 경로를 탄다.
- 이전 형태는 CP에서 `blk_mq_poll_lhp_sleep()`이 `io_poll_delay=-1` 때문에 곧바로 return한 뒤, 실제 `__blk_hctx_poll()` 한 번만 수행했다. 현재 형태는 cookie guard, pre-oneshot poll, `poll_count == UINT_MAX && ret == 1`을 `ret = 0`으로 바꾸는 처리가 추가됐다.
- 그래서 현재 CP 정지는 CP 전용 정책 문제가 아니라, LHP livelock을 잡으려 넣은 공통 `blk_mq_poll_bio()` 변경이 CP의 기존 escape/wakeup 관측 경로까지 건드린 회귀로 보는 것이 가장 강하다.
- 다음 검증은 runtime에서 막힌 fio task가 `__blk_hctx_poll()` 안인지, `bio_poll()` 반복인지, 그리고 `bi_cookie >= q->nr_hw_queues` 또는 `poll_count=UINT_MAX` 조기 반환이 반복되는지 확인하는 것이다.

## LHP 정지 코드 경로 분석
- 현재 `blk_mq_poll_bio()`에서 standalone CP는 `!q->pas_enabled && q->poll_nsec < 0`일 때 pre-oneshot poll과 LHP/PAS sleep 경로를 건너뛴다. 그래서 standalone CP는 `__blk_hctx_poll()` busy-poll 경로만 타며 최근 증상이 줄어든 것으로 본다.
- LHP는 `pas_enabled=0`, `io_poll_delay=0`이므로 standalone CP가 아니고, `blk_mq_poll_lhp_sleep()`까지 들어간다. 이 함수는 `blk_mq_poll_lhp_nsecs()`가 반환한 `q->poll_stat[bucket].mean / 2`만큼 `blk_mq_poll_sleep_nsec()`에서 `TASK_UNINTERRUPTIBLE`로 잔다.
- LHP sleep duration에는 PAS처럼 `BLK_MQ_PAS_MAX_DUR_NS` 상한이 없고, `q->poll_stat`은 queue 전역 상태라 mode 전환/reset knob에서 초기화되지 않는다. 단일 CPU 2T에서 latency sample이 커지거나 이전 실험의 큰 mean이 남으면 LHP가 매우 긴 sleep을 걸어 polled NVMe completion을 늦게 reap해 "막힘"처럼 보일 수 있다.
- 다음 런타임 확인은 막힌 fio task stack/wchan이 `blk_mq_poll_sleep_nsec()`/`io_schedule()`인지, 아니면 `__blk_hctx_poll()`/`nvme_poll()` busy-poll인지 분리하는 것이다.
- 정정: 5.18 LHP도 명시적인 sleep 상한선은 없었다. 다만 5.18 `blk_mq_poll_hybrid()`의 지역 변수 `nsecs`는 `unsigned int`였고, 현재 7.1 이식본의 LHP sleep helper는 `u64 nsecs`라 큰 `poll_stat.mean / 2`가 더 직접적으로 긴 hrtimer sleep으로 전달될 수 있다. 따라서 "상한선 부재"만으로는 5.18 대비 차이를 설명할 수 없고, bio 기반 poll/wait 생명주기와 queue 전역 poll_stat reset 부재를 같이 봐야 한다.
- 정상화 방향 후보: 단일 코어 다중 thread에서는 polled completion을 누군가 계속 reap해야 하므로 LHP/PAS/DPAS sleep duration을 먼저 bounded하게 만들고, queue 전역 `poll_stat`의 mode 간 오염을 차단하며, `blk_mq_poll_bio()`에서 standalone CP뿐 아니라 full-DPAS CP submode도 sleep/pre-hybrid 경로와 분리하는 순서로 보는 것이 맞다. 실제 구현 범위는 사용자 결정이 필요하다.
