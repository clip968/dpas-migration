# 2026-06-28

## 세션 시작 시 현재 위치 복구

- 현재 주제는 단일 CPU에서 CP/polled direct I/O가 멈추는 문제를 `max_no_lock=100` budget으로 해결할지, 5.18식 task-state wakeup 감지를 복원할지 판단하는 것이다.
- 현재 `dpas-kernel/block/blk-mq.c`는 `__blk_hctx_poll()`에서 `q->max_no_lock` 기본값 100을 poll batch budget으로 사용한다.
- `dpas-kernel/block/blk-sysfs.c`의 `pas_max_no_lock` sysfs 최소값은 1이라 런타임에서 budget off 값인 0을 설정할 수 없다.

## 현재 권장 수정 방향

- CP 기본 정책 보존을 우선하면 `max_no_lock` budget은 기본 해결책이 아니라 선택적 안전장치로 내려야 한다.
- `q->max_no_lock` 기본값은 0으로 바꾸고, `pas_max_no_lock` sysfs는 0을 허용해야 한다.
- 단일 CPU 복구는 `__blk_hctx_poll()`에 5.18식 state-aware wakeup 감지를 복원하는 방향으로 잡는다.
- `task_is_running(current)`은 entry state가 `TASK_RUNNING`이 아니었던 경우에만 확인하고, 이 경로로 빠지면 PAS feedback을 막기 위해 `poll_countp = UINT_MAX`로 표시해야 한다.

## 구현 검증 결과

- 현재 `dpas-kernel` 내부 변경은 `max_no_lock` 필드, 초기값, sysfs entry를 제거했지만 `block/blk-mq.c`의 `__blk_hctx_poll()`에는 `READ_ONCE(q->max_no_lock)` 참조가 남아 있다.
- 따라서 현재 구현은 그대로는 빌드 오류가 날 상태이며, `max_poll_count` 변수와 budget break block을 함께 제거해야 한다.
- 또한 5.18식 `stateful_wait`/`task_is_running(current)` wakeup 감지는 아직 구현되어 있지 않다.
- `git diff --check -- block/blk-mq.c block/blk-sysfs.c include/linux/blkdev.h`와 `python3 tools/testing/selftests/dpas/full_mode_switching_static.py`는 통과했지만, macOS 기본 `make 3.81` 때문에 host build object 검증은 시작하지 못했다.

## 재검증 결과

- 이후 재확인 시 `max_no_lock`/`pas_max_no_lock` 잔여 참조는 `dpas-kernel/block`, `dpas-kernel/include`, `tools/testing/selftests/dpas` 범위에서 사라진 것을 확인했다.
- `__blk_hctx_poll()`에는 `stateful_wait` 변수와 TASK_RUNNING 복구는 들어갔지만, 핵심 wakeup 감지 조건인 `task_is_running(current)`가 아직 없다.
- 현재 stateful path는 `signal_pending_state(state, current) || task_sigpending(current)`만 확인하므로, 단일 CPU에서 기대한 5.18식 wakeup escape가 복원됐다고 볼 수 없다.
- `git diff --check`와 DPAS 정적 selftest는 통과했고, object build는 여전히 macOS `make 3.81` 문제로 실행하지 못했다. `gmake`도 현재 PATH에 없다.
