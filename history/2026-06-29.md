## 세션 시작 시 현재 위치 복구

- 현재 주제는 단일 CPU CP/polled direct I/O hang을 `max_no_lock` budget 없이 5.18식 task-state wakeup 감지로 설명하고 검증하는 것이다.
- 현재 `dpas-kernel/block/blk-mq.c`의 `__blk_hctx_poll()`에는 `stateful_wait`, `task_is_running(current)`, `poll_countp = UINT_MAX`, `TASK_RUNNING` 복구가 들어가 있다.
- `max_no_lock`/`pas_max_no_lock` budget 경로는 제거된 상태로 이해한다.

## 오늘 볼 것

- 현재 코드의 요점을 시각적으로 다시 정리한다.
- 핵심은 CP busy poll을 100회 budget으로 끊는 것이 아니라, sleep 상태에서 들어온 poller가 wakeup되었을 때 상위 wait loop로 돌아가게 하는 것이다.

## sync DIO wait loop 이해

- `set_current_state(TASK_UNINTERRUPTIBLE)`은 곧바로 CPU를 재우는 동작이 아니라, completion handler의 wakeup을 놓치지 않도록 "이 task는 기다리는 중"이라고 상태를 먼저 표시하는 단계다.
- 원래 `v7.1-rc4` import에는 이 wait loop와 `blk_io_schedule()`은 있었지만, sync DIO에서 `bio_poll()`로 기다리는 branch는 없었다.
- 현재 sync DIO polled wait branch는 `347554c8d dio의 hipri path 복구`에서 `dio->submit.poll_bio` 추적과 함께 복구된 것이다.

## 4KB random read 단일 CPU CP 흐름

- `scripts/micro_4krr/run_host_dpas_optane.sh`의 CP 모드는 `io_poll=1`, 기본 `io_poll_delay=-1`, `pas_enabled=0`, `switch_enabled=0`이며 fio는 `--bs=4k --rw=randread --ioengine=pvsync2 --iodepth=1 --hipri`로 실행된다.
- 이 조건에서는 XFS sync direct I/O wait loop가 `TASK_UNINTERRUPTIBLE` 상태를 표시한 뒤 `bio_poll()`을 반복 호출한다.
- `pas_enabled=0`이고 `io_poll_delay=-1`이므로 PAS/LHP sleep은 없고, 실제 진행은 `__blk_hctx_poll()`의 driver poll loop가 맡는다.
- 수정 전 문제는 inner poll loop가 completion/need_resched 전까지 CPU를 오래 붙잡아 단일 CPU에서 상위 `iomap` wait loop가 `dio->submit.waiter`를 다시 확인할 기회를 얻기 어려웠다는 점이다.
- 현재 수정은 budget으로 poll을 끊지 않고, sleep state에서 들어온 poller가 `task_is_running(current)`로 wakeup을 감지하면 `poll_countp = UINT_MAX`로 표시하고 상위 loop로 돌아가게 한다.
- 여기서 wakeup은 SSD가 task를 직접 깨우는 것이 아니라, I/O completion path가 `iomap_dio_done()`에서 `dio->submit.waiter = NULL`로 만들고 `blk_wake_io_task(waiter)`를 호출해 기다리던 task를 runnable 상태로 바꾸는 것을 뜻한다.

## 단일 CPU에서 "안 비키면 여전히 안 비키는 것 아닌가" 정리

- 오늘 헷갈렸던 지점은 "수정 전에는 한 thread가 CPU를 안 비켜서 문제였는데, 수정 후에도 단일 CPU에서 poller가 계속 돌면 결국 똑같은 것 아닌가?"였다.
- 결론: 현재 수정은 다른 thread에게 CPU를 양보시키는 코드가 아니다. `bio_poll()` 안에서 이미 completion/wakeup이 관측됐을 때, inner poll loop가 그 사실을 보고 상위 `iomap` wait loop로 돌아가게 하는 코드다.
- 단일 CPU에서도 I/O completion은 일반 userspace thread가 반드시 한 번 더 스케줄되어야만 생기는 것이 아니다.
  - poll path에서는 현재 fio task 자신이 NVMe completion queue를 직접 확인하고 completion을 처리할 수 있다.
  - interrupt/softirq completion은 일반 thread처럼 scheduler time slice를 받아야만 실행되는 것이 아니라, 현재 CPU에서 실행 중인 task를 끊고 들어올 수 있다.
- 따라서 이 문제를 "다른 thread가 못 돌아서 completion이 불가능했다"로만 보면 반쪽짜리 이해다. 더 정확하게는 "completion/wakeup이 발생해 current task state가 `TASK_RUNNING`으로 바뀌어도, 기존 7.1 poll loop가 그 변화를 탈출 조건으로 보지 못했다"가 핵심이다.
- `need_resched()`는 "스케줄러를 돌릴 이유가 생겼는가"를 보는 조건이지, "내 I/O가 완료되어 wakeup되었는가"를 직접 보는 조건이 아니다. 단일 CPU에서 현재 task가 계속 CPU를 잡고 있고 별도 스케줄링 압력이 없으면, wakeup이 있었어도 `need_resched()`만으로는 상위 wait loop로 빨리 돌아가지 못할 수 있다.
- 그래서 5.18식 핵심은 `get_current_state()`로 stateful wait 진입을 알아두고, poll 중 `task_is_running(current)`가 true가 되면 "completion path가 나를 깨웠다"고 보고 즉시 빠져나가는 것이다.

```text
수정 전 이해:
  poller가 CPU를 안 비킨다
    -> 다른 thread가 못 돈다
    -> completion 확인 기회가 늦거나 사라진다

더 정확한 현재 이해:
  iomap wait loop가 current를 TASK_UNINTERRUPTIBLE로 표시한다
    -> bio_poll() / __blk_hctx_poll() 진입
    -> completion path가 dio->submit.waiter = NULL 처리
    -> blk_wake_io_task(waiter)가 current state를 TASK_RUNNING으로 바꿈
    -> 그런데 기존 poll loop는 이 state 변화를 보지 않음
    -> 상위 iomap wait loop로 못 돌아가 waiter == NULL을 확인하지 못함

현재 수정:
  __blk_hctx_poll()이 task_is_running(current)를 확인
    -> wakeup이 있었음을 감지
    -> poll helper return
    -> iomap wait loop가 waiter == NULL 확인
    -> 완료
```

- 따라서 지금 수정의 의미는 "busy poll을 100번마다 강제로 양보한다"가 아니라, "sleep state에서 들어온 poller가 wakeup되었으면 더 이상 CP 의미로 무한히 돌지 말고 완료 조건을 가진 상위 loop로 복귀한다"이다.
