# 2026-06-21

## 세션 시작 시 현재 위치 복구

- 어제 최신 기록은 `history/2026-06-20.md`였고, `history/handoff.md`보다 더 최신이었다.
- 새 host DPAS 커널 `7.1.0-rc4-dpas-host-g63a80d0b9d99`로 부팅했고, `/sys/block/nvme1n1/queue/dpas_switch_stats` 존재와 초기값 0을 확인했다.
- 어제는 host DPAS runner에 debug stats 저장과 parser를 붙였다. 구체적으로 `scripts/micro_4krr/run_host_dpas_optane.sh`는 DPAS run 직후 `dpas_switch_stats`를 저장하도록 바뀌었고, `scripts/micro_4krr/parse_dpas_switch_stats.py`도 추가됐다.
- 이후 `scripts/micro_4krr/run_host_dpas_optane.sh` 실행 중 `nvme1n1 repeat1 PAS 4T RR`에서 fio가 중단됐다.
- 표면 증상은 `fio-3.41`의 `signal=11` 및 `double free or corruption (!prev)`였지만, 사용자 shell의 `dmesg` 확인 결과 같은 시각에 `Oops: divide error`가 찍혔다.
- 전체 Oops trace와 objdump 매핑 기준 문제 위치는 `dpas-kernel/block/blk-mq.c:5625`의 `mul_u64_u64_div_u64(stat->dur, (u64)stat->adj, q->div)` 계산이다.
- 현재 가설은 PAS adaptive duration 계산에서 `stat->dur`/`stat->adj`가 커진 상태로 7.1 이식 코드의 나눗셈 helper를 타면서 x86 `#DE`가 난 것이다. `q->div == 0`인지 quotient overflow인지는 아직 최종 확정하지 않았다.

## 이전 debug stats 수정 상태

- `dpas-kernel/block/blk-sysfs.c`의 `dpas_switch_stats` 출력이 window 진행용 임시 counter가 아니라 누적 stats 필드를 읽도록 수정된 상태다.
- `switch_enabled` write 시 debug stats reset도 `queue_dpas_reset_switch_state()` 안으로 들어가 `dpas_lock` 보호 범위에서 처리되게 했다.
- 이 부분은 정적 검사와 부분 빌드 로그까지는 확인됐지만, VM 최종 `bzImage` 생성 및 VM runtime sysfs 증가/reset 검증은 아직 남아 있다.

## 이어받은 미해결 질문과 리스크

- `PAS 4T RR` 중단의 1차 커널 원인은 `blk_mq_poll_bio()` 안의 PAS duration 계산으로 보이지만, 정확히 `q->div` 0 문제인지 곱셈/나눗셈 quotient overflow인지는 추가 확인이 필요하다.
- 실패 지점은 `blk_mq_poll_bio+0x1b8/0x800`이고, fault 명령은 `divq 0x120(%r9)`로 기록됐다. `request_queue.div` offset `0x120`과 맞아 분모는 `q->div`로 보인다.
- `PAS` 모드에서 터졌으므로 full-DPAS `switch_enabled=1` 전환 로직보다 `switch_enabled=0`에서도 타는 PAS/adaptive duration 경로를 먼저 봐야 한다.
- 이미 완료된 결과는 `1T`의 `INT/CP/LHP/PAS/DPAS`, `4T`의 `INT/CP/LHP/PAS`까지이고, `4T DPAS`와 이후 thread sweep은 실행되지 않았다.
- cleanup 후 `io_poll=1`은 남아 있었으므로 다음 실험 전 의도한 상태인지 확인해야 한다.

## 다음에 볼 것

- `dpas-kernel/block/blk-mq.c`에서 `blk_mq_poll_bio()` 주변 PAS duration update 경로와 `q->div` 초기화/변경 지점을 확인한다.
- 특히 `stat->dur`, `stat->adj`, `q->div`, `cur_case` 갱신 흐름을 확인한다.
- Oops의 전체 register 값, fault instruction, `pahole request_queue` offset 매핑을 다시 묶어 `q->div == 0`과 quotient overflow 가능성을 분리한다.
- 5.18 원본 DPAS 코드와 현재 `mul_u64_u64_div_u64()` 이식 방식 차이를 비교한다.
- 수정 방향은 사용자가 결정해야 한다. 가능한 축은 계산식 overflow-safe 처리, clamp/guard 추가, 또는 5.18 원 코드의 unsigned wrap 동작을 어떻게 재현할지 선택하는 것이다.
- 수정 후에는 host PAS 4T RR smoke test를 먼저 재실행하고, 통과하면 DPAS stats 저장 실험으로 돌아간다.

## 현재 수정 범위 재확인

- `dpas-kernel`의 현재 미커밋 코드 diff는 `block/blk-sysfs.c` 하나뿐이다.
- `blk-mq.c`의 divide fault 지점인 `stat->dur = mul_u64_u64_div_u64(stat->dur, (u64)stat->adj, q->div);` 계산식은 아직 수정되지 않았다.
- `blk-sysfs.c`에는 `queue_dpas_reinit_pas_stats()`를 호출하는 새 `pas_reset_stats` sysfs knob가 추가되어 있다. `1`을 쓰면 `pas_enabled=0` 상태에서 모든 CPU/bucket의 PAS state를 `q->div` 기준으로 재초기화한다.
- 다만 `scripts/micro_4krr/run_host_dpas_optane.sh`에는 아직 `pas_reset_stats` 또는 `pas_d_init`을 매 fio run 전에 호출하는 로직이 없다.
- 기존 `host_smoke_optane.sh`, `host_repeat_optane.sh`, `run_vm.sh` 등에는 `pas_d_init=100`, `pas_up_init=10000`, `pas_dn_init=100000`을 써서 bucket을 초기화하는 패턴이 이미 있다.
