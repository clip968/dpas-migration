## div fault 원인 복구

- fault 위치는 `dpas-kernel/block/blk-mq.c`의 `blk_mq_poll_pas_update_duration()` 안 PAS duration 갱신식이다.
- `q->div`는 queue init에서 `1000000`으로 초기화된다. 기록에 있던 fault 시 `RDX=0x15b245`는 decimal `1421893`으로, 기본 divisor `1000000`보다 크다.
- x86 `divq`는 divisor가 0일 때뿐 아니라 `RDX >= divisor`라 quotient가 `u64`에 안 들어갈 때도 `#DE`를 낸다.
- 따라서 현재 더 강한 원인 가설은 `q->div == 0`보다, PAS adaptive duration에서 `stat->dur`/`stat->adj`가 너무 커져 `mul_u64_u64_div_u64()`의 architecture-specific `divq`가 quotient overflow를 낸 것이다.
- 5.18 원 코드는 `stat[bucket].dur * stat[bucket].adj / q->div` 형태라 64-bit unsigned 곱셈 overflow가 wrap된 뒤 나뉘었지만, 7.1 이식 과정에서 128-bit 곱셈 helper를 쓰면서 overflow trap 조건이 노출됐다.

## 논문 기준 원인 해석

- 원 5.18 코드가 터지지 않은 가장 직접적인 이유는 `stat[bucket].dur * stat[bucket].adj / q->div`가 C의 64-bit unsigned 산술로 먼저 wrap된 뒤 나뉘는 형태였기 때문이다. 잘못된 값이 나올 수는 있어도 x86 `divq` quotient overflow trap으로 이어지지는 않는다.
- 논문은 PAS가 concurrent I/O에서 stale sleep result를 쓰면 sleep duration이 exponential하게 증가할 수 있다고 설명하고, per-core state와 generation-style guard로 이를 줄이려 한다.
- 논문은 CPU contention에서 PAS duration이 0으로 수렴하는 timer failure도 명시한다.
- 다만 논문 본문은 알고리즘/성능 레벨 설명이며, `u64` overflow, C wraparound, x86 `divq #DE`, `mul_u64_u64_div_u64()` overflow trap 같은 구현 산술 문제를 저자가 명시적으로 인지했다고 볼 직접 근거는 찾지 못했다.

## 단일 CPU 실험이 div fault를 드러낸 이유

- 단일 CPU 실험 자체가 직접적인 커널 fault 원인은 아니다. 직접 원인은 큰 `stat->dur * stat->adj / q->div` 계산을 `mul_u64_u64_div_u64()`로 처리하면서 x86 `divq` quotient overflow가 난 것이다.
- 그러나 단일 CPU 실험은 이 문제를 빠르게 드러내는 증폭 조건이다.
- PAS 학습 상태는 전역 bucket 하나가 아니라 `q->pas_stat[cpu][bucket]` 형태의 per-CPU/per-bucket 상태다.
- 4KB random read는 5.18식 mapping 기준 read bucket 6을 주로 사용한다. 여러 CPU에 자연스럽게 분산되면 `CPU0 bucket6`, `CPU1 bucket6`, `CPU2 bucket6`처럼 상태가 CPU별로 나뉘어 진화한다.
- 반대로 `taskset`/`--cpus_allowed`로 fio를 한 CPU에 묶으면 거의 같은 `CPUx bucket6`만 반복해서 사용한다. 이 경우 같은 bucket의 `dur`, `adj`, `sr_last`, `update_req`, `dur_cnt`가 계속 누적된다.
- 따라서 같은 판정, 특히 `UNDER, UNDER`처럼 duration을 늘리는 방향의 피드백이 반복되면 한 bucket의 `adj`/`dur`가 빠르게 커질 수 있다.
- 원래 여러 CPU에 분산될 때는 각 CPU bucket이 받는 업데이트가 나뉘므로 특정 bucket 하나가 같은 속도로 폭주하지 않을 수 있다. 단일 CPU 조건에서는 그 완충이 사라진다.
- 그래서 현재 문제는 "CPU 하나에 몰려서 생긴 정상적인 결과"가 아니라, "CPU 하나에 몰린 실험이 PAS 산술 overflow 취약점을 더 빨리 노출한 것"으로 보는 게 맞다.
- 정상적인 구현이라면 단일 CPU에 몰려도 최악의 경우 PAS duration이 clamp되거나 성능이 나빠지는 수준에서 멈춰야 하며, 커널 Oops까지 가면 안 된다.

## duration clamp 수정 검증

- 사용자가 `blk_mq_poll_pas_scale_duration()` helper를 추가해 `mul_u64_u64_div_u64()` 호출을 직접 scaling helper로 바꾼 상태를 확인했다.
- `rtk git -C dpas-kernel diff --check`는 통과했다.

- diff 대상 `checkpatch.pl --strict`는 아직 실패했다. 남은 항목은 `blk-mq.c`의 helper 호출 줄 정렬, `blk-sysfs.c`의 store 함수 인자 줄 정렬, `if(READ_ONCE(...))`의 공백 누락이다.
- 추가로 `blk-mq.c` 상단에 새로 넣은 `asm-generic/int-ll64.h`, `linux/compiler.h`, `linux/limits.h`, `linux/minmax.h` include는 현재 빌드에는 문제를 만들지 않았지만, 커널 스타일상 불필요해 보이므로 정리 후보로 남긴다.
