# 2026-06-30 DPAS 공부/작업 현황

## 이어받은 맥락
- 전날 `history/2026-06-29-io-uring-kernel-code-path-report.md`를 작성했지만, 현재 질문 흐름은 보고서보다 블록 계층 기초 개념 재정리가 우선이다.
- 현재 헷갈리는 지점은 `bio`가 `request`가 되는 과정, `request_queue`의 역할, merge가 일어날 때 `bio`와 `request`의 관계다.

## 오늘 볼 코드 위치
- `dpas-kernel/block/blk-mq.c`: `blk_mq_submit_bio()`, `blk_mq_bio_to_request()`, `blk_mq_attempt_bio_merge()`
- `dpas-kernel/block/blk-merge.c`: `ll_back_merge_fn()`
- `dpas-kernel/include/linux/blk-mq.h`: `struct request`
- `dpas-kernel/include/linux/blkdev.h`: `struct request_queue`

## 주의할 점
- 일반 blk-mq 개념과 DPAS 전용 hook을 섞어서 설명하지 않는다.
- DPAS는 `request_queue`에 붙은 repo-specific 상태와 정책으로 따로 분리해 설명한다.

## 이번 질문: 5.18 DPAS와 7.1.0-rc4 이식 차이 해석
- 사용자는 "5.18을 그대로 옮긴 게 아니라 per-CPU 상태, sleep loop, QD 추적 생명주기가 바뀌어 DPAS 상태머신이 망가졌다"는 분석의 의미를 물었다.
- 확인한 핵심 위치:
  - `kernel/block/blk-mq.c`: `irq_poll_switch`, `blk_mq_poll_pas_nsecs()`, `blk_mq_poll_hybrid()`
  - `dpas-kernel/block/blk-mq.c`: `blk_mq_poll_sleep_nsec()`, `blk_mq_poll_pas_sleep()`, `blk_dpas_maybe_switch_mode()`
  - `dpas-kernel/include/linux/blkdev.h`: `request_queue`의 `dpas_*` 필드
- 설명할 핵심 직관: 5.18은 "CPU별로, I/O가 sleep loop에 머무는 전체 시간"을 QD로 세고, 7.1 이식본은 "큐 전체에서, single sleep 구간만 짧게" QD로 센다. 따라서 상태머신의 입력값인 평균 QD가 원래 의도와 달라질 수 있다.
- 다음 확인 지점: 실제 수정을 결정하기 전에는 sleep loop 복원, QD 감소 위치 이동, per-CPU 상태 복원/유지 중 어느 범위를 먼저 실험할지 사용자 판단이 필요하다.

## 검증 결과 정정
- per-CPU 상태가 7.1 이식본에서 `request_queue`의 `dpas_*` 상태와 `dpas_lock`으로 옮겨진 것은 코드로 확인된다.
- 5.18의 `blk_mq_poll_hybrid()`에는 `do/while` sleep loop가 있고 7.1 이식본의 `blk_mq_poll_sleep_nsec()`에는 없다. 다만 이 loop는 "timer 만료 후 completion 전까지 계속 재sleep"하는 구조가 아니다. 5.18 코드 주석상 sleep 후 caller가 poll loop를 재시작하고, 미완료면 다음 호출에서 busy poll로 간다.
- 따라서 "5.18은 completion까지 sleep을 유지하므로 T=2 붕괴를 회피한다"는 설명은 코드 근거만으로는 부정확하다. 더 정확한 차이는 5.18이 조기 wakeup이면 원래 sleep deadline까지 다시 자려 하고, 7.1 이식본은 조기 wakeup에도 바로 sleep helper를 빠져나와 poll로 간다는 점이다.
- QD 증가/감소 위치가 달라진 것도 확인된다. 그러나 5.18 QD도 completion까지 유지되는 것이 아니라 hybrid sleep 구간을 중심으로 유지된다. 7.1 이식본의 QD가 실제로 항상 0~1인지, 성능 역전의 주원인이 평가 오버헤드인지 여부는 런타임 로그/벤치로 별도 검증해야 한다.

## 논문 기반 새 가설 검증
- 논문 `fast26-seo/fast26-seo.md` 기준 Figure 9/Section 3.4는 per-device PAS의 stale result/result overwrite 문제와 per-core PAS + Step 3/9 guard를 명시한다.
- Section 3.5는 CPU overload에서 PAS가 scheduler delay를 OVER로 보고 duration을 0으로 줄이는 timer failure를 만들고, DPAS가 timer failure와 QD로 mode switching한다고 설명한다.
- 이 논문 근거는 "단일 CPU 다중 thread에서 DPAS가 완전히 붕괴한다면 알고리즘 자체보다 이식/실험 조건 문제를 먼저 의심해야 한다"를 강하게 뒷받침한다.
- 다만 현재 `dpas-kernel` 코드에는 5.18과 유사한 `update_req`, `dur_cnt`, `dur_cnt_checked` guard가 존재한다. 따라서 "Step 3/9가 빠졌다"가 아니라 "bio/ctx 기반 번역이 최신 poll 구조에서 같은 의미로 작동하는지"를 검증해야 한다.
- 추가 코드 단서: 5.18은 `rq->cpu_num`, `rq->dur_cnt`를 request에 저장해서 sleep한 호출과 completion을 찾는 나중 호출이 달라도 결과를 반영할 수 있다. 7.1은 `struct blk_mq_pas_poll_ctx pas`가 `blk_mq_poll_bio()` stack 변수라서, sleep 직후 같은 호출에서 completion을 못 찾으면 다음 호출에서는 `BIO_LHP_POLL_SLEPT` 때문에 ctx가 active가 아니고 결과 제출이 건너뛰어진다. 이 차이는 Step 3 result submission 유실 가능성을 강하게 만든다.
- 현재 checkout에는 `BLK_MQ_PAS_MAX_DUR_NS 100000ULL` clamp가 있으므로, 10ms/5s sleep 폭주 데이터가 clamp 이전 커널인지, requested sleep이 아니라 I/O latency인지 확인이 필요하다.
- 2026-06-27 기록의 CPU0 5회 결과는 PAS 2T 이상 약 119~120 kIOPS, DPAS 4T 이상 OL/INT 전환으로 남아 있어, 사용자가 새로 제시한 "0.2 IOPS/10ms" 데이터와 같은 실험인지 확인해야 한다.

## 수정 방향 도출
- 1차 수정 후보는 sleep loop 복원이 아니라, 5.18의 `request` 기반 PAS sleep 결과 메타데이터를 7.1의 `bio_poll()` 생명주기에 맞게 `bio`에 보존하는 것이다.
- 근거: 5.18은 `rq->cpu_num`, `rq->dur_cnt`, `rq->dur`가 completion을 잡는 나중 poll 호출까지 남지만, 7.1은 `struct blk_mq_pas_poll_ctx pas`가 `blk_mq_poll_bio()`의 stack 변수라서 `__blk_hctx_poll()`이 `ret == 0`으로 끝나면 결과 제출 상태가 사라진다.
- 권장 형태: `struct bio`에 작은 PAS poll context를 추가하고, `bio_init()`/`bio_reset()`에서 초기화하며, `blk_mq_poll_pas_complete()`가 `ret <= 0`일 때 context를 지우지 않고 다음 `bio_poll()` 호출에서 completion 결과를 반영하게 만든다.
- 이 변경은 `struct bio` 크기 증가와 block layer 구조 변경을 포함하므로, 실제 구현은 사용자 승인 후 진행해야 한다.

## before/after 설명 포인트
- 사용자에게 실제 수정 지점을 before/after 형식으로 설명한다.
- 수정 지점은 4곳으로 제한한다: `blk_types.h`의 `struct bio`, `bio.c`의 `bio_init()/bio_reset()`, `blk_mq_poll_pas_sleep()`, `blk_mq_poll_pas_complete()/blk_mq_poll_bio()`.
- 설명의 핵심은 "stack-local `pas`를 없애고 `bio->bi_pas`에 보존한다"이다. `ret <= 0`이면 아직 completion을 못 본 것이므로 context를 유지하고, `ret > 0`에서 결과를 반영한 뒤 clear한다.

## 외부 수정안 검토
- 사용자가 제시한 "7.1에서는 cookie에서 rq를 얻을 수 없으므로 `blk_mq_poll_bio()`에 sleep+poll 루프를 넣고 `BIO_LHP_POLL_SLEPT`를 제거하자"는 안을 검토했다.
- `cookie`로 `rq`를 얻을 수 없다는 판단은 맞지만, 5.18의 `blk_mq_poll_hybrid()`도 timer 만료 후 completion이 없으면 caller가 poll loop를 재시작하고 다음 호출은 busy poll로 가도록 주석이 명시한다. 따라서 "5.18은 한 호출 안에서 completion까지 재sleep한다"는 전제는 부정확하다.
- `BIO_LHP_POLL_SLEPT`는 7.1에서 5.18의 `RQF_MQ_POLL_SLEPT`에 대응하는 one-sleep guard로 보는 것이 더 자연스럽다. 이를 제거하고 `bio_poll()` 내부에 completion까지 반복 sleep을 넣으면 5.18 호환 복원이 아니라 새 알고리즘이 된다.
- 1차 권장안은 그대로 `bio->bi_pas` 보존 패치다. QD 감소 지점 이동과 sleep 루프 추가는 별도 실험 후보로 분리해야 하며, 첫 패치에 섞으면 원인 분리가 어렵다.

## 사용자 수정본 리뷰
- 사용자가 `bio->bi_pas` 보존 패치를 대체로 적용했다. `blk_mq_pas_poll_ctx` stack 구조체 제거, `bio_init()` 초기화, `blk_mq_poll_pas_sleep()`의 `bio->bi_pas` 기록, `blk_mq_poll_bio()` 호출부 변경은 의도와 맞다.
- 남은 문제: `blk_mq_poll_pas_complete()`에서 `ret > 0` 처리 후 `bio->bi_pas.active`를 clear하지 않는다. CPU mismatch 경로도 clear 없이 return한다. 이러면 stale PAS context가 같은 bio의 이후 poll에 남아 잘못된 결과 제출 또는 불필요한 mode switch를 유발할 수 있다.
- `bio_reset()`은 `BIO_RESET_BYTES == offsetof(struct bio, bi_max_vecs)`이고 `bi_pas`가 `bi_max_vecs` 앞에 있으므로 memset으로 초기화된다. 별도 명시 초기화는 필요하지 않지만 helper로 만들면 가독성은 좋아진다.
- `git -C dpas-kernel diff --check`는 통과했다. 빌드는 아직 수행하지 않았다.
- 이후 사용자 수정으로 CPU mismatch 경로와 정상 completion 처리 후 `ctx->active = false`가 추가됐다. 1차 `bio->bi_pas` 보존 패치의 핵심 동작은 들어간 상태다.
- 남은 것은 커널 빌드와 T=2/T=4 런타임 로그 검증이다. 주석 스타일은 kernel style에 맞게 다듬는 정도의 사소한 정리만 남아 있다.
- `blk_mq_poll_pas_complete()`의 `ret <= 0` 보존 주석을 kernel block-comment 스타일로 정리했다. 동작 변경은 없다.

## 수정 필요성 설명
- 단일 CPU 다중 스레드에서 `__blk_hctx_poll()`은 `need_resched()` 때문에 completion을 못 잡고 `ret == 0`으로 끊겨 나올 가능성이 커진다.
- 기존 7.1 포팅본은 PAS sleep metadata가 `blk_mq_poll_bio()`의 stack 변수였기 때문에, `ret == 0`으로 한 번 return하면 sleep 결과 판정에 필요한 `cpu/bucket/dur_cnt`가 사라졌다.
- 다음 `bio_poll()` 호출에서는 `BIO_LHP_POLL_SLEPT` 때문에 재sleep이 없고 stack ctx도 새로 active되지 않는다. 그 결과 completion을 나중에 잡아도 `sr_last/update_req`가 갱신되지 않아 PAS/DPAS가 CPU contention을 OVER/timer-failure로 학습하지 못할 수 있다.
- 이번 수정은 그 metadata를 `bio->bi_pas`에 저장해 여러 `bio_poll()` 호출 사이에 유지시키는 것이다.
