- 단, INT/CP/PAS도 평균 IOPS가 `+15%~+19%` 올랐다. 따라서 warmup 효과를 LHP 내부 효과로만 해석하면 안 된다.
- warmup 결과의 순위는 `CP > PAS ~= LHP > INT`이다. PAS는 LHP보다 IOPS가 약 `1.27%` 높고, LHP는 PAS보다 CPU total이 낮다.

### 추가 확인: warmup 비교 해석 주의

- measured run과 warmup run 모두 `--direct=1`, 4KiB random read, `pvsync2`, `iodepth=1`이다.
- no-warmup과 warmup의 큰 IOPS 차이는 평균 latency뿐 아니라 tail latency 차이에서 왔다.
  - 예: CP repeat-1 p95 latency가 no-warmup `35.584us`, warmup `8.384us`.
  - CP는 PAS/LHP sleep 로직이 없는 busy-poll 기준이므로, 이 차이는 LHP/PAS 알고리즘 효과가 아니다.
- 현재 가장 그럴듯한 해석은 warmup이 workload를 바꾼 것이 아니라, 같은 파일/LBA 범위에 대해 measured run 직전 direct random read pass를 추가해 filesystem metadata, device/controller, CPU/scheduler 상태를 precondition했다는 것이다.
- 따라서 기존 두 로그 비교만으로 "warmup이 LHP latency를 낮췄다"라고 결론 내리면 안 된다. warmup 조건은 "cold measured run"이 아니라 "same-region direct random read 후 측정"에 가깝다.

## Raw block preconditioning probe 준비

- 사용자가 ext4 metadata 효과를 분리하기 위해 raw block read-only 실험을 진행하기로 했다.
- Codex 세션에서는 `/dev/nvme1n1p1` 접근 권한이 없고, `sudo -n true`도 interactive authentication required로 실패했다. 따라서 root 터미널에서 실행할 스크립트를 준비했다.
- 새 스크립트: `scripts/micro_4krr/raw_block_precondition_probe.sh`
- 중요한 추가 변인: `fio --cmdhelp=randrepeat` 확인 결과 기본값은 `randrepeat=1`, `randseed=0x89`이다.
  - 기존 warmup/measured run은 명시하지 않았지만 같은 random offset sequence를 반복했을 가능성이 있다.
  - raw probe는 이 변인을 명시적으로 통제한다.
- 기본 probe는 `CP`만 실행한다. CP는 LHP/PAS sleep duration bucket 통계를 쓰지 않으므로, filesystem/device/CPU preconditioning 효과를 먼저 분리하기 좋다.
- raw probe case:
  - `cold`: warmup 없이 measured run.
  - `same-seed`: warmup과 measured가 같은 seed `0x89`를 사용한다. 같은 raw offset sequence 반복 효과를 확인한다.
  - `diff-seed`: warmup은 `0x89`, measured는 `0x12345`를 사용한다. 같은 100MiB raw range 안에서 다른 random sequence를 읽었을 때 tail이 줄어드는지 본다.
- 공통 통제: `/dev/nvme1n1p1` read-only, `--readonly`, `--rw=randread`, `--direct=1`, `--bs=4k`, `--ioengine=pvsync2`, `--iodepth=1`, `--numjobs=1`, `runtime/ramp=27s/3s`, warmup `5s/3s`, raw offset/size `4g/100m`, 각 run 전 `drop_caches`.

## Optane 원본 스타일 CP CPU 62% 원인 분석

- 질문: 원본-style CP 1T가 busy polling 기대치인 CPU 100% 근처가 아니라 `usr 10.51% + sys 51.61% = 62.12%`, context switch `2,879,044`로 나온 이유.
- 핵심 결론:
  - 원본-style CP 결과는 "classic busy polling CP"로 보기 어렵다.
  - fio에는 `--hipri`가 들어가 polling 경로는 켜졌지만, sysfs knob 상태는 CP가 아니라 PAS/LHP 계열 상태를 물려받았을 가능성이 높다.
- 근거:
  - `run_optane_full_dpas_no_ehp.sh`의 mode 순서는 `PAS INT CP LHP`.
  - 현재 host에서 `/sys/module/nvme/initstate`가 없어서 스크립트의 `nvme_setup()`은 modprobe reload를 skip한다.
  - 이 경우 PAS가 `io_poll_delay=0`, `pas_enabled=1`, `pas_adaptive_enabled=1`을 설정한 뒤 INT/CP가 이를 reset하지 않는다.
  - 현재 read-only sysfs 확인도 `io_poll=1`, `io_poll_delay=0`, `pas_enabled=1`, `pas_adaptive_enabled=1`, `poll_queues=20`.
  - host-repeat CP는 `set_mode()`가 매번 `io_poll_delay=-1`, `pas_enabled=0`, `pas_adaptive_enabled=0`으로 reset한 뒤 실행했고, CP 1T repeat 평균은 CPU `99.99%`, context switch 평균 `76`.
  - 원본-style CP 1T는 issued IO `2,879,116`과 context switch `2,879,044`가 거의 1:1이라, busy loop라기보다 거의 매 I/O마다 sleep/wakeup이 발생한 패턴이다.
- 커널 코드 근거:
  - `BLK_MQ_POLL_CLASSIC=-1`.
  - `io_poll_delay=-1`이면 `q->poll_nsec < 0`으로 sleep 없이 classic polling.
  - `io_poll_delay=0`이면 adaptive LHP sleep 계산 경로로 들어간다.
  - `pas_enabled=1`이면 `blk_mq_poll_bio()`에서 LHP가 아니라 PAS sleep 경로를 먼저 탄다.
- 다음 확인:
  - 파괴적 작업 없이 `cat /sys/module/nvme/initstate`, `cat /sys/block/nvme1n1/queue/{io_poll,io_poll_delay,pas_enabled,pas_adaptive_enabled,nomerges}`, `cat /sys/module/nvme/parameters/poll_queues`만 확인하면 된다.
