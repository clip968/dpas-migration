# 2026-05-31 DPAS 현황판

## 현재 위치

새 서버에서 기존 DPAS VM 테스트 환경을 `dpas-kernel/` 최신 `main` 기준으로 재현했다. 대상은 5.18 `kernel/` 트리가 아니라 `dpas-kernel/` 7.1.0-rc4 기반이다.

현재 `dpas-kernel` HEAD는 다음 커밋이다.

```text
19009d1c0 fixed lhp 구현
```

## 이어받은 작업

- out-of-tree build dir는 `build/dpas-kernel-vm`이다.
- VM용 `.config`는 `/boot/config-7.0.0-15-generic` 기반으로 만들었다.
- initramfs 없는 직접 `bzImage` 부팅을 위해 `virtio_blk`, `ext4`, `nvme` 관련 필수 항목을 built-in으로 맞췄다.
- Ubuntu 패키징 전용 인증서 경로인 `debian/canonical-certs.pem`, `debian/canonical-revoked-certs.pem` 참조는 VM용 build config에서 비웠다.
- Debian cloud image를 root backing image로 사용했고, VM domain은 `dpas-kernel-7-1-0-rc4`로 만들었다.
- ISO cloud-init seed는 initramfs 없는 부팅에서 `iso9660`이 module이라 읽히지 않았다. 그래서 built-in VFAT으로 seed를 다시 만들었다.
- 최종 root overlay는 `build/libvirt-vm/dpas-root-vfat.qcow2`, seed는 `build/libvirt-vm/dpas-seed-vfat.img`이다.

- `bzImage` 산출물: `build/dpas-kernel-vm/arch/x86/boot/bzImage`

- `/sys/block/nvme0n1/queue/io_poll`: `1`
- `/sys/module/nvme/parameters/poll_queues`: `1`
- guest `fio`는 `/usr/bin/fio`, 버전은 `fio-3.39`이다.
- 같은 VM 환경을 다시 만들기 위한 절차와 시행착오는 `history/2026-05-31-dpas-vm-reproduction-guide.md`에 정리했다.

## 편집기 clangd 설정

Zed/clangd가 `dpas-kernel/block/blk-mq.c`에서 커널 include path와 generated headers를 몰라 가짜 에러를 내는 문제를 줄이기 위해 `dpas-kernel/compile_commands.json`을 다시 생성했다.

- 기존 `compile_commands.json`은 `[]`만 담긴 빈 DB였다.
- 현재 작업공간에는 예전 `build/dpas-kernel-vm` 산출물과 `/boot/config-7.0.0-15-generic`이 없어서, 현재 WSL의 `/proc/config.gz`를 기반으로 `build/dpas-kernel-vm/.config`를 다시 만들었다.
- `make -C dpas-kernel O=../build/dpas-kernel-vm olddefconfig` 후 `block/blk-mq.o`를 빌드해 `.cmd` 파일과 generated headers를 만들었다.
- `python3 scripts/clang-tools/gen_compile_commands.py -d ../build/dpas-kernel-vm -o compile_commands.json`로 `dpas-kernel/compile_commands.json`을 생성했다.
- 이후 전체 커널 소스에 가까운 clangd DB를 만들기 위해 `make -k -C dpas-kernel O=../build/dpas-kernel-vm -j8`을 실행했다. 현재 WSL config 기준으로 `bzImage`와 모듈 빌드까지 exit 0으로 끝났고, `.cmd` 파일은 13,331개 생성됐다.
- 최종 `dpas-kernel/compile_commands.json`은 9,091개 compile command, 고유 파일 9,087개를 담는다. 이 중 `dpas-kernel/` source tree 파일은 8,137개, build generated 파일은 950개다.
- 관심 경로인 `block/blk-core.c`, `block/blk-mq.c`, `block/blk-sysfs.c`, `io_uring/io_uring.c`, `fs/xfs/xfs_super.c`, `drivers/nvme/host/core.c`, `drivers/nvme/host/pci.c`, `drivers/nvme/host/ioctl.c` 엔트리가 모두 포함된다.
- `compile_commands.json` 크기는 약 33MiB이고, out-of-tree build dir `build/dpas-kernel-vm`은 약 14GiB까지 커졌다.

주의: 이 설정은 편집기/clangd용 메타데이터다. 현재 WSL config 기반이라 기존 VM 부팅 검증에 사용한 `/boot/config-7.0.0-15-generic` 기반 config와 완전히 같은 VM 재현 상태를 의미하지는 않는다. 또한 커널의 `compile_commands.json`은 현재 `.config`에서 실제 빌드되는 파일 중심이므로, Kconfig에서 꺼진 드라이버/아키텍처 파일까지 문자 그대로 전부 포함한다는 뜻은 아니다.

## fixed LHP smoke 검증

`/mnt/nvme-test/dpas-fio-testfile` 256MiB 파일을 준비한 뒤, guest에서 `fio pvsync2 + direct=1 + randread + runtime=2s`로 짧게 경로 검증만 했다. 성능 평가는 하지 않았다.

- negative baseline: `hipri=0`, `io_poll_delay=-1`, `pas_enabled=0`
  - `bio_poll`, `blk_mq_poll_bio`, `blk_hctx_poll`, `nvme_poll`, LHP sleep 지점, PAS increment 모두 0.
- classic polling baseline: `hipri=1`, `io_poll_delay=-1`, `pas_enabled=0`
  - `bio_poll=28243`, `blk_mq_poll_bio=28243`, `blk_hctx_poll=28243`, `nvme_poll=6064287`.
  - LHP sleep 지점과 PAS increment는 0.
- fixed LHP: `hipri=1`, `io_poll_delay=10`, `pas_enabled=0`
  - FIO exit 0.
  - `bio_poll=27862`, `blk_mq_poll_bio=27862`, `blk_hctx_poll=27862`, `nvme_poll=3357776`.
  - inlined `blk_mq_poll_lhp_sleep()` sleep 진입점인 `blk_mq_poll_bio+0xd6` count는 27862, 실제 `io_schedule()` 지점인 `+0x164` count는 27858.
  - `dmesg --level=emerg,alert,crit,err,warn` 기준 테스트 전후 diff 없음.
- PAS hook 회귀:
  - `pas_enabled=0`, `hipri=1`: polling path 유지, `pas_increment=0`.
  - `pas_enabled=1`, `hipri=1`: polling path 유지, `pas_increment=27972`.

테스트 후 `/sys/block/nvme0n1/queue/io_poll_delay=-1`, `pas_enabled=0`으로 원복했다.

### kprobe timestamp 기반 sleep duration 확인

코드 수정 없이 `blk_mq_poll_bio+0x164`(`io_schedule()` call 직전)과 `blk_mq_poll_bio+0x169`(`io_schedule()` return 직후)에 kprobe를 걸고 trace timestamp 차이를 계산했다.

- 설정: `io_poll_delay=10`, `pas_enabled=0`, `hipri=1`, `fio pvsync2 + direct=1 + randread`, runtime 1초.
- kprobe hit: `lhp_sleep_before=14295`, `lhp_sleep_after=14295`.
- paired samples: 14295.
- trace timestamp delta: min 6.000us, avg 13.696us, p50 14.000us, p90 14.000us, p99 21.000us, max 138.000us.
- FIO exit 0, dmesg WARN/error diff 없음.

이 값은 `io_schedule()` call 직전부터 return 직후까지의 wall-clock delta라, hrtimer sleep 시간에 wakeup/scheduler overhead가 포함된다.

## adaptive PAS 진입 논의와 코드 정리

fixed LHP smoke 이후 다음 단계가 adaptive PAS인지 확인했다. 결론은 "PAS로 넘어갈 기준선은 갖췄지만, PAS 완성은 아니다"로 정리했다.

현재 완료로 보는 기준선:

- sync DIO + HIPRI 경로가 살아 있다.
- `bio_poll()` -> `blk_mq_poll_bio()` -> `blk_hctx_poll()` -> `nvme_poll()` 경로가 runtime에서 확인됐다.
- fixed LHP sleep path가 `io_schedule()` 지점까지 실제로 도달했다.
- `pas_enabled` no-sleep checkpoint hook 회귀가 없었다.

아직 PAS 완성으로 보지 않는 이유:

- bucket별 `dur` 선택과 update가 runtime에서 아직 검증되지 않았다.
- `sr_last` / `sr_pnlt` 기반 UNDER/OVER 판정이 실제 workload에서 돌았는지 아직 보지 않았다.
- `dur_cnt`, `dur_cnt_checked`, `update_req`가 중복 update를 막는지 runtime에서 확인해야 한다.

### 개념 정리

PAS bucket은 I/O를 read/write 방향과 size로 나눠 sleep duration을 따로 학습하기 위한 슬롯이다. 5.18 artifact 기준 mapping은 read가 짝수 bucket, write가 홀수 bucket을 쓴다.

```text
bucket = ddir + 2 * ilog2(sectors)

4KB read  -> sectors=8, ddir=0 -> bucket=6
4KB write -> sectors=8, ddir=1 -> bucket=7
```

`update_req`는 "새 sleep 결과가 생겼고, 아직 `dur`에 반영하지 않았다"는 pending flag로 정리했다.

```text
poll 결과 발생
-> oversleep/undersleep 판정
-> sr_last 갱신
-> update_req = 1

다음 같은 bucket 사용 전
-> update_req == 1 확인
-> sr_pnlt/sr_last 조합으로 adj 계산
-> dur 갱신
-> dur_cnt 증가
-> update_req = 0
```

여기서 result는 latency 숫자가 아니라 binary classification이다.

```text
poll_count == 0 -> oversleep
poll_count > 0  -> undersleep
```

### 코드 수정 상태

`dpas-kernel/block/blk-mq.c`에 adaptive PAS 최소 경로를 정리했다.

- `init_pas_stat()`에 PAS bucket 초기값 설명 주석을 추가했다.
- `__blk_hctx_poll()`을 만들고 `poll_count`를 선택적으로 받아올 수 있게 했다.
- 기존 `blk_hctx_poll()`은 wrapper로 유지해 기존 caller 동작을 보존했다.
- fixed LHP와 PAS가 같이 쓰는 `blk_mq_poll_sleep_nsec()` 공통 sleep helper를 만들었다.
- `blk_mq_poll_pas_bucket()`으로 `bio` 기준 5.18식 bucket index를 계산한다.
- `blk_mq_poll_pas_update_duration()`에서 `update_req`가 있으면 `adj`와 `dur`를 갱신하고 `update_req=0`으로 내린다.
- `blk_mq_poll_pas_sleep()`에서 `q->pas_stat[cpu][bucket].dur`를 snapshot한 뒤 실제 hrtimer sleep을 수행한다.
- `blk_mq_poll_pas_complete()`에서 poll 결과를 보고 `sr_last`, `dur_cnt_checked`, `update_req`를 갱신한다.
- `task_sigpending()` 때문에 completion 없이 `1`이 반환되는 경우는 `UINT_MAX` sentinel로 표시해 PAS result로 반영하지 않게 했다.

주의할 점:

- `get_cpu()` 구간에서는 per-cpu PAS state snapshot만 잡고, 실제 sleep은 `put_cpu()` 이후에 수행한다.
- `BIO_LHP_POLL_SLEPT` guard를 그대로 써서 같은 `bio`에 대한 중복 sleep을 막는 방향이다.
- `pas_adaptive_enabled`로 `up/dn` 자체를 동적으로 바꾸는 DPAS1/DPAS2식 확장은 아직 넣지 않았다. 지금은 bucket `dur` update core만 들어간 상태다.

## 남은 리스크

- SSH IP는 libvirt DHCP lease라 VM 재생성/네트워크 변경 시 달라질 수 있다.
- 이번 guard 확인은 kprobe count 기준으로 LHP sleep count가 `bio_poll` count를 넘지 않는다는 수준이다. 같은 bio가 여러 번 `bio_poll()`되는 workload에서 `BIO_LHP_POLL_SLEPT`가 중복 sleep을 막는지까지는 별도 bio pointer 추적이 필요하다.
- adaptive PAS 코드는 build까지 통과했지만, VM runtime에서 `pas_enabled=1`으로 `dur`, `dur_cnt`, `sr_last`, `update_req`가 실제로 변하는지는 아직 확인하지 않았다.
- 현재 PAS result 판정은 `poll_count <= q->poll_threshold`를 oversleep으로 본다. 이 threshold가 5.18 재현과 현재 `bio_poll()` 경로에서 충분한지 runtime 관찰이 필요하다.

## 다음 작업

1. 새 `bzImage`로 VM을 부팅하고 `uname -r`이 새 빌드인지 확인한다.
2. `pas_enabled=0`에서 fixed LHP/classic polling 회귀가 없는지 짧게 재확인한다.
3. `pas_enabled=1`, `hipri=1`에서 FIO smoke를 돌리고 `bio_poll -> blk_mq_poll_bio -> blk_hctx_poll -> nvme_poll` 경로가 유지되는지 본다.
4. kprobe 또는 임시 계측으로 `blk_mq_poll_pas_update_duration()`, `blk_mq_poll_pas_complete()` 진입과 bucket별 `dur_cnt/update_req/sr_last` 변화를 확인한다.
5. 같은 bio 재-poll 상황을 강제로 만들 필요가 있으면 kprobe fetcharg 또는 임시 tracepoint로 bio pointer별 PAS/LHP sleep 횟수를 본다.
6. 성능 평가는 아직 하지 않는다. 먼저 adaptive PAS core가 crash 없이 돌고, duration update가 실제로 관찰되는지 확인한다.
