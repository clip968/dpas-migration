# 2026-06-04 DPAS 현황판

## 현재 위치

이전 작업은 fixed LHP smoke 검증을 지나 adaptive PAS 최소 경로를 `dpas-kernel/`에 도입한 지점에서 멈췄다.

- 현재 상위 repo `main`: `a898ca899 history update`
- 현재 `dpas-kernel/` `main`: `8e7968051 adaptive pas 도입`
- 대상 커널 트리: 5.18 `kernel/`이 아니라 Linux 7.1.0-rc4 기반 `dpas-kernel/`
- 주요 코드 위치: `dpas-kernel/block/blk-mq.c`

## 이어받은 검증 상태

- fixed LHP는 guest runtime smoke에서 `hipri=1`, `io_poll_delay=10`, `pas_enabled=0` 기준 `blk_mq_poll_bio()` sleep 경로와 `io_schedule()` 지점까지 도달하는 것이 확인됐다.
- kprobe timestamp 기준 fixed sleep duration도 짧게 측정했다.
- adaptive PAS 최소 코드는 build/static 검증까지 끝났다.
- 새 adaptive PAS `bzImage`를 VM에 싣고 `pas_enabled=1` runtime smoke를 돌리는 검증은 아직 하지 않았다.

## 남은 리스크

- VM SSH IP는 libvirt DHCP lease라 바뀔 수 있다.
- `BIO_LHP_POLL_SLEPT` guard가 같은 `bio` 재-poll 상황에서 중복 sleep을 실제로 막는지는 bio pointer 단위로 아직 확인하지 않았다.
- adaptive PAS의 `dur`, `dur_cnt`, `sr_last`, `update_req` 변화는 VM runtime에서 아직 관찰하지 않았다.
- 현재 PAS result 판정인 `poll_count <= q->poll_threshold`가 7.1 `bio_poll()` 경로에서 충분한지는 runtime 관찰이 필요하다.

## 다음에 볼 것

1. 새 `bzImage`로 VM을 부팅하고 `uname -r`이 `8e7968051` 계열 새 빌드인지 확인한다.
2. `pas_enabled=0`에서 classic polling/fixed LHP 회귀가 없는지 짧게 재확인한다.
3. `pas_enabled=1`, `hipri=1`에서 FIO smoke를 돌려 polling path가 유지되는지 본다.
4. kprobe 또는 임시 계측으로 `blk_mq_poll_pas_update_duration()`, `blk_mq_poll_pas_complete()` 진입과 bucket별 state 변화를 확인한다.
5. 성능 평가는 아직 하지 않는다. 먼저 adaptive PAS core가 crash 없이 돌고 duration update가 실제로 관찰되는지 확인한다.
