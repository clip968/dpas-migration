## 2026-06-04 BIO_LHP_POLL_SLEPT guard 검증 (kprobe, 코드 수정 없음)

위 "남은 것" 1번을 닫았다. 같은 bio 재-poll 시 중복 sleep을 guard가 실제로 막는 것을 bio/sector 단위로 확인했다.

### 재-poll 경로 확인

- fio가 ext4 파일을 쓰므로 sync DIO는 iomap 경로다. `dpas-kernel/fs/iomap/direct-io.c:881-891`은 upstream과 달리 `REQ_POLLED` bio면 `bio_poll()` 실패 시 `blk_io_schedule()` 없이 **같은 bio로 bio_poll()을 계속 재호출**하도록 수정되어 있다 (`347554c8d` "dio의 hipri path 복구" 추정).
- 따라서 `__blk_hctx_poll()`이 completion 없이 반환하면 같은 bio가 `blk_mq_poll_pas_sleep()`에 다시 들어오고, 두 번째 hrtimer sleep을 막는 유일한 방어선이 `BIO_LHP_POLL_SLEPT`다.

### 계측

기존 probe에 fetcharg 추가:

- `poll_entry`: `blk_mq_poll_bio` entry, `bio=%si`, `lhp=+0x15(%si):u8`(bi_flags 상위 바이트, guard bit=0x20), `sector=+0x28(%si):u64`(bi_iter.bi_sector, 인스턴스 식별자)
- `sleep_in`/`sleep_ret`: `blk_mq_poll_sleep_nsec` entry/return
- bio pointer는 slab 재사용으로 매번 같은 주소가 나오므로 randread의 sector로 인스턴스를 구분했다.

### 재-poll 유도 시행착오 (중요한 부수 발견)

1. idle: poll_entry == sleep == 28014, 재-poll 0회. guard 차단 경로는 idle에서 한 번도 실행되지 않는다.
2. 같은 CPU에 CFS busy loop (`taskset -c 1`): IOPS 14.2k->10k로 경쟁은 실재했지만 **재-poll 여전히 0회**.
3. RT waker (`chrt -f 10`, 1ms 주기): avg lat 70->176us로 영향은 컸지만 **재-poll 여전히 0회**.
4. 원인: 이 커널은 `CONFIG_PREEMPT_LAZY=y` + `PREEMPT_DYNAMIC`이고 runtime 모드가 `lazy`다 (`/sys/kernel/debug/sched/preempt`는 full/lazy만 제공, voluntary 전환 불가).
   - CFS 경쟁자의 선점 요구는 `TIF_NEED_RESCHED_LAZY`라 `need_resched()`에 안 보이고,
   - RT wakeup의 즉시 flag는 인터럽트 복귀에서 바로 선점되어 poll 루프가 확인하기 전에 소비된다.
   - 즉 **이 커널 기본 설정에서 `__blk_hctx_poll()`의 `need_resched()` 탈출 경로는 사실상 죽은 코드**다. 5.18 시절 voluntary 커널과 다른 동작이므로 PAS 의미론 검토 시 기억할 것.
5. 성공한 유도: `task_sigpending()` 탈출 경로. fio에 SIGUSR1 폭풍(3ms 주기, cpu2에서 발사)을 보내면 `TIF_SIGPENDING`이 syscall 복귀까지 유지되어 poll이 1을 반환하고 iomap 루프가 같은 bio로 재진입한다. fio는 USR1을 stats 출력으로 처리해서 죽지 않는다.

### 결과 (signal storm run, fio err=0)

- `poll_entry` 28252 = fresh 26906(`lhp=1`) + **재-poll 1346(`lhp=33`, guard bit 0x20 set)**
- `sleep_in` = `sleep_ret(=1)` = 26906. **재-poll 1346회에서 sleep 시도 0회** — guard가 `blk_mq_poll_pas_sleep()` 입구(5522)에서 전부 차단.
- 인스턴스(sector) 단위 분석: 총 26906개, 재-poll 발생 108개(인스턴스당 최대 76회), **"sleep은 정확히 1회" 위반 0건**, 인스턴스 내 sector 불일치 0건.
- 부수 확인: `pas_complete_upd` = 26798 = 26906 - 108. **재-poll이 발생한 인스턴스는 PAS result 갱신이 누락**된다(sleep한 호출은 sigpending/0으로 끝나고, completion을 찾은 재-poll 호출은 ctx inactive). sigpending UINT_MAX sentinel도 의도대로 동작. 재-poll이 잦은 환경에서는 PAS 학습이 정체된다는 뜻이고, 이는 contention에서 hybrid polling이 약하다는 논문 관찰 및 mode switching 동기와 직결된다.

### 부수 발견: bucket state 수명

- PAS bucket state는 연속 run 사이, 그리고 `pas_enabled` 0->1 toggle 후에도 유지된다 (toggle은 재초기화하지 않음).

## 2026-06-04 PAS/DPAS1-2 개념 정리와 다음 수정 위치

이번 세션은 코드 수정을 바로 넓히기보다, 현재 PAS core가 어떤 상태값으로 움직이는지 먼저 풀어서 이해하는 데 집중했다. 특히 `dur_cnt`, `dur_cnt_checked`, `update_req`, `cur_case`, DPAS1/DPAS2의 `up/dn` 조정 의미를 정리했다.

### `dur_cnt`, `dur_cnt_checked`, `update_req`

현재 초기화 위치는 `dpas-kernel/block/blk-mq.c:init_pas_stat()`이다.

```c
stat->dur_cnt = 1;
stat->dur_cnt_checked = 0;
stat->update_req = 0;
```

정리한 의미:

- `dur_cnt`는 처리한 IO 개수가 아니라 **현재 bucket `dur`의 세대 번호**다. `dur`가 갱신될 때마다 증가한다.
- `dur_cnt_checked`는 **현재 세대의 sleep 결과가 completion 쪽에서 이미 반영 처리됐는지**를 표시한다.
- `update_req`는 **직전 sleep + poll 결과가 생겼고, 아직 다음 `dur` 갱신에 소비되지 않았다**는 pending flag다.

상태 흐름:

```text
초기:
dur_cnt=1, dur_cnt_checked=0, update_req=0

1번 세대 dur로 sleep:
ctx.dur_cnt=1 저장

completion 성공:
dur_cnt=1, dur_cnt_checked=1, update_req=1

다음 sleep 진입:
update_req를 소비해서 dur 갱신
dur_cnt=2, dur_cnt_checked=1, update_req=0
```

따라서 `dur_cnt_checked`가 항상 `dur_cnt - 1`인 것은 아니다.

- 새 세대가 시작됐고 아직 completion 결과가 없으면 `dur_cnt_checked == dur_cnt - 1`처럼 보인다.
- completion 결과가 기록되어 다음 update를 기다리는 동안은 `dur_cnt_checked == dur_cnt`가 정상이다.

또한 다른 IO라도 같은 `dur`를 쓰면 같은 `dur_cnt`를 가질 수 있다. 즉 `dur_cnt`는 IO별 unique id가 아니라 "이 IO가 어떤 duration 세대로 잤는가"를 구분하는 tag다.

### `cur_case` 계산

현재 PAS duration update는 다음 식으로 최근 두 번의 sleep 결과를 2-bit case 번호처럼 만든다.

```c
cur_case = stat->sr_pnlt * 2 + stat->sr_last;
```

여기서 현재 코드 주석 기준으로:

```text
0 = overslept
1 = underslept
```

따라서 가능한 조합은 네 개뿐이다.

| case | `sr_pnlt` | `sr_last` | 의미 |
|---|---:|---:|---|
| 0 | 0 | 0 | overslept -> overslept |
| 1 | 0 | 1 | overslept -> underslept |
| 2 | 1 | 0 | underslept -> overslept |
| 3 | 1 | 1 | underslept -> underslept |

`case 0/3`은 같은 판정이 연속된 상태다. 조정이 아직 부족하다는 뜻으로 보고 `heat_up`을 적용해 조정폭을 키운다.

`case 1/2`는 판정 방향이 바뀐 상태다. 목표 근처에서 흔들린다고 보고 `cool_dn`을 적용해 조정폭을 줄인다.

### DPAS1/DPAS2의 차이

`pas_adaptive_enabled` 값의 의미:

```text
0: DPAS1/DPAS2 adaptive up/dn 조정 꺼짐
1: DPAS1
2: DPAS2
```

DPAS1:

- `dn`을 adaptive하게 조정한다.
- `up = dn / updn_ratio`로 맞춘다.
- 즉 `up/dn`이 같이 움직인다.
- 더 공격적으로 반응하지만 진동이 커질 수 있다.

DPAS2:

- `up`만 adaptive하게 조정한다.
- `dn`은 고정 쪽에 가깝게 둔다.
- 더 보수적인 실험 모드다.

`up`을 `dn`보다 작게 두는 이유도 정리했다. `dn`은 oversleep일 때 duration을 줄이는 조정폭이고, `up`은 undersleep일 때 duration을 늘리는 조정폭이다. oversleep은 IO가 이미 끝났는데도 늦게 깨는 것이므로 latency 손실이 직접적이다. 그래서 duration을 줄이는 `dn`은 크게 두고, duration을 늘리는 `up`은 보수적으로 둔다. 기본 비율은 `up = dn / 10`이다.

예시:

```text
dn = 100000 -> 10% 감소
up = 10000  -> 1% 증가
```

즉 PAS는 "너무 오래 자는 것"을 더 위험하게 보고 빠르게 줄이며, "너무 짧게 자는 것"은 CPU 낭비 문제라 천천히 늘리는 정책이다.

### 현재 코드 수정 상태

현재 `dpas-kernel/block/blk-mq.c` working tree에는 `blk_mq_poll_pas_update_duration()` 뒤쪽에 DPAS1/DPAS2 `case 0/3` heat-up 블록이 추가된 상태다. 아직 커밋되지 않았다.

추가된 형태:

```c
stat->dur_cnt++;

if (q->pas_adaptive_enabled) {
	if (cur_case == 0 || cur_case == 3) {
		if (q->pas_adaptive_enabled == 1) {
			/* DPAS 1: adjust dn, derive from dn */
			stat->dn = stat->dn * (q->div + q->heat_up) / q->div;
			if (stat->dn > q->max_dn)
				stat->dn = q->max_dn;

			stat->up = stat->dn / q->updn_ratio;
		} else if (q->pas_adaptive_enabled == 2) {
			/* DPAS 2: adjust up only */
			stat->up = stat->up * (q->div + q->heat_up) / q->div;
			if (stat->up > q->div / 10)
				stat->up = q->div / 10;
		}
	}
}
```

이 블록의 의미:

- `case 0/3`은 같은 판정이 연속된 상태다.
- DPAS1이면 `dn`을 `heat_up` 비율만큼 키우고 `up = dn / updn_ratio`로 다시 맞춘다.
- DPAS2이면 `up`만 `heat_up` 비율만큼 키운다.
- `q->div = 1000000`, `q->heat_up = 50000`이면 `(div + heat_up) / div = 1.05`라서 5% 증가를 fixed-point 정수 계산으로 표현한 것이다.

### 아직 빠진 부분

현재 추가된 코드는 `case 0/3` heat-up만 처리한다. 5.18 원본 DPAS1/DPAS2와 맞추려면 `case 1/2` cool-down 블록도 이어서 넣어야 한다.

다음에 넣을 형태:

```c
else {
	stat->up = stat->up * (q->div - q->cool_dn) / q->div;

	if (q->pas_adaptive_enabled == 1) {
		if (stat->up < q->min_dn / q->updn_ratio)
			stat->up = q->min_dn / q->updn_ratio;

		stat->dn = stat->up * q->updn_ratio;
	} else if (q->pas_adaptive_enabled == 2) {
		if (stat->up < q->div / 10000)
			stat->up = q->div / 10000;
	}
}
```

의미:

- `case 1/2`는 over/under 방향이 바뀐 상태라 목표 근처에서 진동 중이라고 본다.
- `cool_dn`으로 `up`을 줄인다.
- DPAS1이면 줄어든 `up`을 기준으로 `dn = up * updn_ratio`를 다시 맞춘다.
- DPAS2이면 `up`만 줄이고 `dn`은 유지한다.

### DPAS1/2 이후의 큰 단계

DPAS1/2를 이식하고 테스트하면 PAS 내부 duration controller는 더 완성된다. 그 다음은 full DPAS mode switching이다.

구분:

```text
DPAS1/2:
  PAS mode 안에서 up/dn 조정폭을 adaptive하게 바꾸는 기능

full DPAS mode switching:
  classic polling / PAS / overloaded / interrupt mode 사이를 workload에 따라 오가는 policy layer
```

주의할 점:

- 5.18 원본은 `_CP`, `_PAS`, `_OL`, `_INT` mode와 `switch_param*` 기반 전환 로직을 갖고 있다.
- 최신 `dpas-kernel`에서는 `IOCB_HIPRI`면 submission side에서 `REQ_POLLED`가 붙고, 이 경우 처음부터 `HCTX_TYPE_POLL`로 갈 수 있다.
- 따라서 full interrupt mode는 poll loop에서 `bio_poll()`을 안 부르는 것만으로 충분하지 않을 수 있다.
- 나중에 진짜 interrupt mode를 구현하려면 submission side에서 `REQ_POLLED`를 붙일지 말지도 함께 검토해야 한다.

다음 작업 순서:

1. 현재 heat-up 블록에 이어 `case 1/2` cool-down 블록을 추가한다.
2. `pas_adaptive_enabled=0/1/2` 각각에서 `up/dn/dur/dur_cnt` 변화가 예상대로 다른지 kprobe로 확인한다.
3. DPAS1/2 검증이 끝난 뒤에야 full mode switching 설계를 시작한다.
4. 성능 평가는 여전히 그 다음 단계다.
