# 2026-06-04 DPAS 현황판

## 현재 위치

이전 작업은 fixed LHP smoke 검증을 지나 adaptive PAS 최소 경로를 `dpas-kernel/`에 도입한 지점에서 멈췄다.

- 현재 상위 repo `main`: `a898ca899 history update`
- 현재 `dpas-kernel/` `main`: `8e7968051 adaptive pas 도입`
- 대상 커널 트리: 5.18 `kernel/`이 아니라 Linux 7.1.0-rc4 기반 `dpas-kernel/`
- 주요 코드 위치: `dpas-kernel/block/blk-mq.c`

## 이어받은 검증 상태

- fixed LHP는 guest runtime smoke에서 `hipri=1`, `io_poll_delay=10`, `pas_enabled=0` 기준 `blk_mq_poll_bio()` sleep 경로와 `io_schedule()` 지점까지 도달하는 것이 확인됐다.
- kprobe timestamp 기준 fixed sleep duration도 짧게 측정했다.
- adaptive PAS 최소 코드는 build/static 검증까지 끝났다.
- 새 adaptive PAS `bzImage`를 VM에 싣고 `pas_enabled=1` runtime smoke까지 통과했다.

- `BIO_LHP_POLL_SLEPT` guard가 같은 `bio` 재-poll 상황에서 중복 sleep을 실제로 막는지는 bio pointer 단위로 아직 확인하지 않았다.

## 2026-06-04 adaptive PAS 내부 계측 결과 (kprobe, 코드 수정 없음)

위 "다음에 볼 것" 1, 2번과 4번 일부를 코드 수정 없이 kprobe로 확인했다.

### 계측 방법

- guest `/proc/kallsyms`에서 `blk_mq_poll_pas_sleep/complete/update_duration`과 `blk_mq_poll_lhp_sleep`은 심볼이 없었다. 전부 `blk_mq_poll_bio()`에 inline됐다.
- `blk_mq_poll_sleep_nsec`(공통 sleep helper)과 `__blk_hctx_poll`은 심볼이 살아 있다.
- host `build/dpas-kernel-vm/vmlinux`를 `objdump -d -l --disassemble=blk_mq_poll_bio`로 disassemble해서 source line이 붙은 offset을 잡았다. KASLR은 symbol+offset 방식 kprobe라 영향 없다.
- `struct blk_rq_pas_stat` 레이아웃은 disassembly와 일치: `dur`+0x00(u64), `adj`+0x08, `up`+0x10, `dn`+0x18, `sr_pnlt`+0x20, `sr_last`+0x21, `update_req`+0x22, `dur_cnt`+0x23, `dur_cnt_checked`+0x24, stride 40B.

kprobe 지점 (`8e7968051` 빌드 기준, 함수 base `blk_mq_poll_bio`):

| probe | offset | 의미 | fetcharg |
|---|---|---|---|
| `pas_branch` | +0x9f | PAS branch 진입 (inlined `pas_sleep` 시작) | - |
| `pas_stat` | +0x141 | per-cpu `stat[bucket]` 접근 직후 | `%si`=&stat[bucket]에서 전체 state, `%r13`=bucket*40 |
| `pas_upd` | +0x14c | `update_req==1` 확인 후 dur 갱신 body | - |
| `pas_complete_entry` | +0x231 | complete가 ret/poll_count/pas_stat guard 통과 | - |
| `pas_complete_upd` | +0x2d9 | complete에서 `sr_last`/`update_req=1` 실제 store | `%cx`=poll_count, `%dx`=&stat[bucket] |
| `lhp_branch` | +0x27e | fixed LHP branch 진입 | - |
| `sleep_in`/`sleep_ret` | `blk_mq_poll_sleep_nsec` 심볼 | 실제 hrtimer sleep | `%si`=nsecs, `$retval` |

주의: 이 offset들은 `8e7968051` + 현재 `.config` 빌드 전용이다. 재빌드하면 다시 objdump로 잡아야 한다.

### Test A: adaptive PAS 전체 사이클 (`pas_enabled=1`, `hipri=1`, 3초 FIO)

### Test B: cold-start 성장 곡선

`pas_d_init`에 store하면 `queue_dpas_reinit_pas_stats()`가 모든 bucket을 재초기화한다(`echo 100 > pas_d_init`이 코드 수정 없는 reset knob).

## 2026-06-04 BIO_LHP_POLL_SLEPT guard 검증 (kprobe, 코드 수정 없음)

위 "남은 것" 1번을 닫았다. 같은 bio 재-poll 시 중복 sleep을 guard가 실제로 막는 것을 bio/sector 단위로 확인했다.

### 재-poll 경로 확인

- fio가 ext4 파일을 쓰므로 sync DIO는 iomap 경로다. `dpas-kernel/fs/iomap/direct-io.c:881-891`은 upstream과 달리 `REQ_POLLED` bio면 `bio_poll()` 실패 시 `blk_io_schedule()` 없이 **같은 bio로 bio_poll()을 계속 재호출**하도록 수정되어 있다 (`347554c8d` "dio의 hipri path 복구" 추정).
- 따라서 `__blk_hctx_poll()`이 completion 없이 반환하면 같은 bio가 `blk_mq_poll_pas_sleep()`에 다시 들어오고, 두 번째 hrtimer sleep을 막는 유일한 방어선이 `BIO_LHP_POLL_SLEPT`다.

### 계측

기존 probe에 fetcharg 추가:

- `poll_entry`: `blk_mq_poll_bio` entry, `bio=%si`, `lhp=+0x15(%si):u8`(bi_flags 상위 바이트, guard bit=0x20), `sector=+0x28(%si):u64`(bi_iter.bi_sector, 인스턴스 식별자)
- `sleep_in`/`sleep_ret`: `blk_mq_poll_sleep_nsec` entry/return
- bio pointer는 slab 재사용으로 매번 같은 주소가 나오므로 randread의 sector로 인스턴스를 구분했다.

### 재-poll 유도 시행착오 (중요한 부수 발견)

1. idle: poll_entry == sleep == 28014, 재-poll 0회. guard 차단 경로는 idle에서 한 번도 실행되지 않는다.
2. 같은 CPU에 CFS busy loop (`taskset -c 1`): IOPS 14.2k->10k로 경쟁은 실재했지만 **재-poll 여전히 0회**.
3. RT waker (`chrt -f 10`, 1ms 주기): avg lat 70->176us로 영향은 컸지만 **재-poll 여전히 0회**.
4. 원인: 이 커널은 `CONFIG_PREEMPT_LAZY=y` + `PREEMPT_DYNAMIC`이고 runtime 모드가 `lazy`다 (`/sys/kernel/debug/sched/preempt`는 full/lazy만 제공, voluntary 전환 불가).
   - CFS 경쟁자의 선점 요구는 `TIF_NEED_RESCHED_LAZY`라 `need_resched()`에 안 보이고,
   - RT wakeup의 즉시 flag는 인터럽트 복귀에서 바로 선점되어 poll 루프가 확인하기 전에 소비된다.
   - 즉 **이 커널 기본 설정에서 `__blk_hctx_poll()`의 `need_resched()` 탈출 경로는 사실상 죽은 코드**다. 5.18 시절 voluntary 커널과 다른 동작이므로 PAS 의미론 검토 시 기억할 것.
5. 성공한 유도: `task_sigpending()` 탈출 경로. fio에 SIGUSR1 폭풍(3ms 주기, cpu2에서 발사)을 보내면 `TIF_SIGPENDING`이 syscall 복귀까지 유지되어 poll이 1을 반환하고 iomap 루프가 같은 bio로 재진입한다. fio는 USR1을 stats 출력으로 처리해서 죽지 않는다.

### 결과 (signal storm run, fio err=0)

- `poll_entry` 28252 = fresh 26906(`lhp=1`) + **재-poll 1346(`lhp=33`, guard bit 0x20 set)**
- `sleep_in` = `sleep_ret(=1)` = 26906. **재-poll 1346회에서 sleep 시도 0회** — guard가 `blk_mq_poll_pas_sleep()` 입구(5522)에서 전부 차단.
- 인스턴스(sector) 단위 분석: 총 26906개, 재-poll 발생 108개(인스턴스당 최대 76회), **"sleep은 정확히 1회" 위반 0건**, 인스턴스 내 sector 불일치 0건.
- 부수 확인: `pas_complete_upd` = 26798 = 26906 - 108. **재-poll이 발생한 인스턴스는 PAS result 갱신이 누락**된다(sleep한 호출은 sigpending/0으로 끝나고, completion을 찾은 재-poll 호출은 ctx inactive). sigpending UINT_MAX sentinel도 의도대로 동작. 재-poll이 잦은 환경에서는 PAS 학습이 정체된다는 뜻이고, 이는 contention에서 hybrid polling이 약하다는 논문 관찰 및 mode switching 동기와 직결된다.

### 부수 발견: bucket state 수명

- PAS bucket state는 연속 run 사이, 그리고 `pas_enabled` 0->1 toggle 후에도 유지된다 (toggle은 재초기화하지 않음).
- `pas_d_init` store는 **모든 CPU**의 bucket을 재초기화한다. 반면 학습은 fio가 도는 CPU의 bucket에서만 진행되므로, unpinned fio로 학습시킨 뒤 다른 CPU에 pin해서 돌리면 cold-start처럼 보인다. 테스트 시 `--cpus_allowed`로 CPU를 고정하고 시작 전 상태를 명시할 것.

### 사후 상태

- kprobe 전부 제거, trace buffer 원복, preempt 모드는 건드리지 않음(lazy 유지).
- 게스트 최종 상태: `pas_enabled=0`, `io_poll=1`, `io_poll_delay=-1`.
- `dmesg`에 BUG/Oops/panic/WARNING 없음.
