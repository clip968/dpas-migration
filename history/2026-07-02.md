# 2026-07-02 DPAS 공부/작업 현황

## 이어받은 맥락
- 현재 이슈는 LHP가 계속 멈추거나 끝나지 않는 현상 때문에, 이전에 덜 위험했던 커널로 롤백할지 현재 커널에서 계속 디버깅할지 판단하는 것이다.
- root repo는 `dpas-kernel` submodule pointer가 `9193bdd03`에서 `ee97b44cf`까지 이동한 상태이고, `dpas-kernel` 내부 작업트리는 clean이다.
- 최근 LHP/CP 관련 변경은 모두 `dpas-kernel/block/blk-mq.c`에 집중돼 있다.

## 확인한 기준점
- 2026-06-06 host Optane 4-mode smoke 기준 커널은 `7.1.0-rc4-dpas-host-g4be3fefb1311`이고, `INT/CP/LHP/PAS` 20개 fio run 모두 `err=0`, 새 NVMe timeout/reset/panic/oops 없음으로 기록돼 있다.
- 이 지점까지 되돌리는 것은 안정 기준선으로는 강하지만, 현재 full DPAS/PAS duration/bio ctx/livelock 관련 작업을 크게 잃는 넓은 롤백이다.
- 최근 회귀 격리 기준점은 `9193bdd03` 이후다. `9193bdd03..ee97b44cf` 차이는 `block/blk-mq.c` 약 40줄이며, cookie guard, pre-oneshot poll, standalone CP guard, LHP/PAS sleep 호출 순서가 핵심이다.

## 현재 판단
- 지금 바로 `4be3fefb` 계열로 완전 롤백하는 것은 최종 복구 수단으로 남겨두고, 먼저 `9193bdd03` 또는 그 직후 커밋 단위로 되돌린 테스트 커널을 만들어 LHP 증상이 최근 `blk_mq_poll_bio()` 변경 때문인지 확인하는 쪽이 더 낫다.
- 현재 코드에서 LHP는 `pas_enabled=0`, `io_poll_delay=0`이라 `standalone_cp`가 아니고, pre-oneshot poll 뒤 completion을 못 잡으면 `blk_mq_poll_lhp_sleep()`으로 들어간다.
- LHP sleep 시간은 `q->poll_stat[bucket].mean / 2`이고 별도 상한이 없다. 큰 latency sample이나 이전 실험의 queue 전역 poll_stat이 남으면 단일 CPU에서 polled completion reaping을 늦춰 멈춤처럼 보일 수 있다.
- 다만 LHP가 실제로 `blk_mq_poll_sleep_nsec()`/`io_schedule()`에서 자는지, 아니면 `__blk_hctx_poll()`/`nvme_poll()` busy-poll에서 못 빠지는지는 아직 런타임 stack으로 분리되지 않았다.

## 다음 확인 지점
- 막힌 fio task의 `wchan`/stack을 확인해 LHP sleep 정지인지 busy-poll 반복인지 먼저 분리한다.
- 같은 workload로 `ee97b44cf`, `932425da8`, `bd3df84d9`, `9193bdd03`를 커밋 단위로 테스트하면 최근 변경 중 어디서 LHP가 깨지는지 좁힐 수 있다.
- 시간이 급하면 `4be3fefb` host 커널은 known-less-risk 기준선으로 부팅해 데이터를 확보할 수 있지만, 그 결과만으로 현재 코드의 root cause가 사라지는 것은 아니다.

## LHP/PAS sleep-first 교체
- 사용자가 제시한 diff 방향대로 `dpas-kernel/block/blk-mq.c`의 `blk_mq_poll_bio()`에서 hctx-level pre-oneshot poll을 제거했다.
- 새 흐름은 `pas_enabled = READ_ONCE(q->pas_enabled)`, `classic_poll = !pas_enabled && READ_ONCE(q->poll_nsec) < 0`을 먼저 snapshot하고, classic CP가 아닐 때만 LHP/PAS sleep을 먼저 수행한 뒤 실제 `__blk_hctx_poll()`을 한 번 호출한다.
- 이유는 hctx-level pre-poll이 같은 hctx를 공유하는 다른 bio의 completion을 잡을 수 있으므로, 그 결과로 현재 bio의 LHP/PAS delay를 건너뛰게 하면 sleep-before-poll 정책이 깨질 수 있기 때문이다.
- `blk_mq_poll_pas_sleep()`의 `BLK_POLL_ONESHOT` 조기 return 제거는 현재 작업트리에 이미 반영돼 있었다.
- 기대 효과는 CP는 delay 없이 classic poll만 수행하고, LHP/PAS는 completion 확인 전에 먼저 delay를 수행해 정책 의미를 단순하게 복구하는 것이다.
