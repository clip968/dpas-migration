# 2026-05-27 DPAS 현황판

## 현재 위치

오늘의 중심은 Part 4다.

현재 Part 4의 실제 진행 방향은 예전의 "PAS-only skeleton만 먼저 올린다"에서 다음 순서로 바뀌었다.

```text
sync DIO HIPRI path 복구
-> 5.18 LHP 포팅
-> PAS 포팅
-> cp/int/lhp/ehp/pas mode 비교 및 정리
```

오늘 이해해야 할 핵심은 이것이다.

```text
DPAS 5.18 실험 경로를 최신 커널에서 재현하려면
filesystem sync DIO + IOCB_HIPRI가 REQ_POLLED를 타고
bio_poll() -> blk_mq_poll() -> nvme_poll()까지 내려가야 한다.
```

## 오늘 한 작업의 의미

### request_queue 직접 필드 방식으로 방향 전환

처음에는 최신 커널에 맞춰 `q->dpas` 같은 분리 구조를 생각했지만, 사용자가 5.18 방식 재현을 우선하기로 결정했다.

그래서 현재 방향은 다음과 같다.

```text
새로운 추상화 우선 설계
-> 보류

5.18처럼 request_queue에 DPAS/PAS 상태를 직접 둠
-> 현재 우선 방향
```

이 결정의 의미:
- 5.18 코드와 비교하기 쉽다.
- 논문 실험 경로를 먼저 재현하기 좋다.
- 나중에 upstream-style cleanup은 따로 해야 한다.
- 지금은 "깔끔한 설계"보다 "5.18과 같은 동작 경로 확보"가 우선이다.

### DPAS/PAS sysfs knob 복구

`pas_enabled`, `pas_adaptive_enabled`, `ehp_enabled`, `switch_param1~7` 같은 값들은 단순 설정값이 아니라 5.18 DPAS 동작을 제어하던 실험 인터페이스다.

현재 의미:
- 아직 모든 mode switching이 살아난 것은 아니다.
- 하지만 5.18식 실험 파라미터를 받을 그릇은 다시 생겼다.
- `pas_d_init`, `pas_up_init`, `pas_dn_init`은 값만 저장하면 부족하고, per-cpu `pas_stat` bucket 재초기화와 연결되어야 한다.

### sync DIO HIPRI path 복구

최신 커널의 `fs/iomap/direct-io.c`는 sync DIO에서 polling을 막고 있었다.

기존 구조의 의미:

```text
IOCB_HIPRI가 있어도 sync DIO이면 bio_set_polled()를 하지 않음
-> REQ_POLLED가 붙지 않음
-> HCTX_TYPE_POLL로 가지 않음
-> bio_poll() / blk_mq_poll() 경로를 밟지 않음
```

현재 수정 방향의 의미:

```text
IOCB_HIPRI이면 bio를 REQ_POLLED로 표시
-> poll 대상 bio를 dio->submit.poll_bio에 저장
-> submit loop 이후 iocb->private에도 연결
-> sync wait loop에서 bio_poll() 직접 호출
```

이 작업은 단순히 조건문 하나를 지운 것이 아니다. submit path와 wait path를 같이 맞춘 것이다.

## 오늘 이해한 핵심 개념

### 1. polling I/O는 "보내는 방식"과 "기다리는 방식"이 한 세트다

interrupt 기반 I/O:

```text
bio 제출
-> task 잠듦
-> device interrupt
-> task wake-up
```

polling 기반 I/O:

```text
bio 제출
-> REQ_POLLED 표시
-> poll queue로 감
-> caller가 bio_poll()을 직접 호출
-> completion을 직접 찾음
```

따라서 `REQ_POLLED`만 붙이고 기다리는 쪽이 `blk_io_schedule()`만 하면 위험하다. polling으로 보냈으면 polling으로 기다려야 한다.

### 2. 최신 커널이 sync DIO polling을 막은 이유

sync DIO polling이 원리적으로 불가능해서 막은 것이 아니다.

문제는 최신 sync DIO wait loop가 polling completion을 직접 진행시키지 않는 구조였기 때문이다.

문제 상황:

```text
REQ_POLLED로 제출됨
-> interrupt completion을 기대하면 안 됨
-> 그런데 sync wait loop는 잠들기만 함
-> 아무도 bio_poll()을 호출하지 않음
-> timeout/hang 위험
```

그래서 최신 upstream은 sync DIO HIPRI를 막고, polling의 중심을 io_uring 쪽으로 옮겼다.

### 3. DPAS가 지금 최신 커널 방향을 일부 역행하는 이유

DPAS 5.18 실험은 `pvsync2 + hipri + direct=1` 같은 legacy sync DIO 경로를 전제로 한다.

최신 커널 방향:

```text
sync DIO HIPRI 제한
io_uring polling 중심
```

DPAS 재현 방향:

```text
sync DIO HIPRI 복구
REQ_POLLED 복구
bio_poll() wait loop 복구
그 위에 LHP/PAS hook 포팅
```

즉, 이것은 upstream 품질의 일반화가 아니라 5.18 논문 실험 경로를 재현하기 위한 연구용 복구다.

### 4. `dio->submit.poll_bio`와 `iocb->private`의 차이

둘은 같은 bio를 가리킬 수 있지만 목적이 다르다.

```text
dio->submit.poll_bio
  sync DIO 내부 wait loop가 직접 bio_poll()할 때 쓰는 포인터

iocb->private
  외부 iopoll 함수인 iocb_bio_iopoll()이 bio를 찾을 때 쓰는 포인터
```

그래서 5.18식으로 보면 구조는 이렇게 잡는 것이 자연스럽다.

```text
submit 중:
  dio->submit.poll_bio에 poll 대상 bio 저장

submit loop 이후:
  iocb->private에 dio->submit.poll_bio 연결

sync wait loop:
  dio->submit.poll_bio로 bio_poll()

async iopoll:
  iocb->private로 bio_poll()
```

### 5. `bio->bi_cookie`는 request tag가 아니다

오늘 다시 정리한 polling path에서 중요한 점:

```text
bio->bi_cookie
  poll 대상 hctx 번호로 사용됨
  request tag 자체가 아님
```

요청이 시작될 때 polled bio의 cookie에는 `rq->mq_hctx->queue_num`이 저장된다. 이후 `bio_poll()`은 이 cookie를 이용해 어느 hardware context를 poll할지 찾는다.

## 질문과 답

### Q. 5.18에는 CONFIG_DPAS 같은 켜고 끄는 개념이 있었나?

있었다. 5.18 DPAS는 실험 기능을 켜고 끄는 개념이 있었고, sysfs knob들도 runtime 제어를 위해 존재했다.

현재는 최신 커널식으로 완전히 재설계하기보다 5.18과 같은 필드와 knob를 먼저 복구하는 방향이다.

### Q. sync DIO HIPRI를 살린다는 것은 io_uring이 아니라 기본 read/write도 살리는 것인가?

맞다. 여기서 말하는 복구는 io_uring 전용 polling이 아니라 `O_DIRECT + RWF_HIPRI` 기반 sync read/write도 block polling path로 다시 들어가게 하는 것이다.

### Q. 그러면 사실상 최신 커널 방향을 역행하는 구조인가?

맞다. 다만 목적은 upstream 반영이 아니라 5.18 DPAS 실험 경로 재현이다.

### Q. `!is_sync_kiocb(iocb)`만 제거하면 되는가?

아니다. 그것은 bio를 polling queue로 보내는 문만 여는 것이다.

필수로 같이 해야 하는 것:
- poll bio 저장
- submit 이후 `iocb->private` 연결
- sync wait loop에서 `bio_poll()` 호출

### Q. `iocb->private`에 bio를 바로 저장하지 않아도 되는가?

sync DIO wait loop만 보면 바로 저장하지 않아도 된다. 하지만 async iopoll 경로를 보존하려면 `iocb->private` 연결 자체는 유지되어야 한다.

오늘 정리한 방향은 submit 중에는 `dio->submit.poll_bio`만 정하고, submit loop가 끝난 뒤 `iocb->private`에 한 번 연결하는 방식이다.

## 헷갈리기 쉬운 점

### sync DIO가 polling을 못하는 것이 아니다

정확히는 "현재 최신 커널의 sync DIO wait 구조에서는 그냥 켜면 안전하지 않다"가 맞다.

### `REQ_POLLED`는 완료를 자동으로 만들어주지 않는다

`REQ_POLLED`는 요청을 poll 가능한 queue로 보내는 표시다. 완료를 실제로 찾아오는 것은 `bio_poll()` / `blk_mq_poll()` 호출이다.

### io_uring을 무시하면 안 된다

지금 첫 목표는 pvsync2/sync DIO 경로지만, 최신 커널의 hybrid polling과 io_uring 경로는 나중에 반드시 비교해야 한다. 특히 PAS/LHP sleep을 넣을 때 double sleep 위험이 있다.

### Part 4 부모 페이지와 현재 실제 방향이 다르다

Part 4 부모 페이지에는 여전히 PAS-only minimal port 문맥이 강하게 남아 있다. 하지만 최신 하위 계획은 sync DIO/LHP-first로 수정되었다. 답변하거나 작업할 때는 반드시 현재 child plan을 기준으로 봐야 한다.

## 남은 리스크

- 현재 sync DIO HIPRI 복구는 코드 형태를 맞춘 단계다. 실제 실행 중 polling path가 안정적으로 도는지는 아직 별도 확인이 필요하다.
- 최신 upstream이 피한 sync DIO polling 위험을 다시 여는 것이므로 hang/timeout 가능성을 계속 의식해야 한다.
- `direct-io.c`에 들어간 설명 주석은 kernel style에 맞게 짧은 block comment로 정리해야 한다.
- LHP/PAS sleep을 붙이기 전, sync DIO path와 io_uring path의 역할 차이를 분명히 해야 한다.
- request_queue 직접 필드 방식은 5.18 재현에는 좋지만, 장기적으로는 cleanup 비용이 생긴다.
