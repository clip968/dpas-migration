## CP 제외 sweep 시작 중 precondition 지연

- 사용자가 `Starting 32 processes`에서 오래 걸린다고 보고했다.
- 확인 결과 이는 본측정이 아니라 `scripts/xfs_fio_prerun.sh` precondition 단계의 `fio --numjobs=32 --runtime=60` write-fill이다.
- `sudo taskset -c 0 ... run_host_dpas_optane.sh`로 실행했기 때문에 이 precondition도 CPU0 하나에 묶인다.
- 동시에 이전 run의 `fio --numjobs=2 --cpus_allowed=0 --hipri` CP 프로세스가 아직 살아 있어 CPU0과 같은 device를 경합하고 있었다.
- 따라서 이 run의 결과는 폐기하고, 이전 fio/runner를 정리한 뒤 다시 시작하는 것이 맞다. `dmesg`는 Codex 세션에서 sudo TTY가 없어 직접 확인하지 못했다.

## CP 제외 sweep 중 2T/LHP hang 관찰

- 사용자가 `LHP`도 `CP`와 같이 `2T`에서 너무 오래 걸린다고 보고했다.
- 관찰 시점의 fio는 `--numjobs=2 --cpus_allowed=0 --cpus_allowed_policy=split --hipri`였고, `scripts/micro_4krr/fio_data/nvme1n1/RR/2T/LHP/fio_report_1.log`는 `Starting 2 processes`까지만 있는 141B 파일이었다.
- queue knob는 `io_poll=1`, `io_poll_delay=0`, `pas_enabled=0`, `pas_adaptive_enabled=0`, `switch_enabled=0`으로 순수 LHP 상태였다.
- 현재 증거만으로 deadlock은 확정하지 않는다. deadlock은 서로 lock/resource를 기다리는 상태인데, 지금은 task stack과 `dmesg` 확인이 필요하다.
- 다만 10초 runtime을 넘겨 진행이 없으므로 이 run은 hang으로 취급하고 중단 후 `dmesg`와 task stack을 확인하는 것이 맞다.
- 이후 사용자가 붙여준 stack에서 PID 6369는 `hrtimer_nanosleep -> common_nsleep -> __x64_sys_clock_nanosleep`에 있었고, block/NVMe poll 경로 stack은 캡처되지 않았다.
- 붙여준 `dmesg --level=err,warn`에는 해당 시각 block/NVMe/Oops/hung task 메시지가 없고 UFW 로그가 대부분이었다. 따라서 현재 증거는 커널 Oops나 명확한 lock deadlock보다는 fio/user-space 쪽 sleep 또는 아직 캡처되지 않은 worker 상태 확인 필요 쪽이다.

## 여러 CPU full runner 결과 분석

- 사용자는 단일 CPU 대신 여러 CPU를 허용해 `INT,CP,LHP,PAS,DPAS`, thread `1,2,4,8,16,20`, repeat 5 결과를 생성했다.
- 로그 배치는 fio report 150개와 DPAS stats 30개로 맞고, 결과 파싱 파일은 `scripts/micro_4krr/result_data/nvme1n1-RR-repeat_5.txt`와 `scripts/micro_4krr/result_data/nvme1n1-RR-dpas-switch-repeat_5.txt`다.
- 여러 CPU에서는 `CP`와 `LHP`가 모두 `2T`를 포함해 완주했다. 단일 CPU hang은 여러 CPU를 열면 재현되지 않는다.
- IOPS는 16T/20T에서 모든 mode가 약 582k로 포화된다. 이 구간은 IOPS보다 CPU 사용량 차이가 핵심이다.
- CP는 모든 thread에서 CPU가 약 100%다. LHP는 약 52~58% 수준이고, PAS/DPAS는 높은 thread에서 더 낮다.
- DPAS는 1T에서 `pas_to_cp`/`cp_to_pas`가 많이 발생해 CP처럼 동작하고, 2T/4T에서는 `pas_stay_not_qd1`이 많아 PAS에 머문다.
- DPAS는 8T 이상에서 `pas_to_ol=1`, `ol_to_int`/`int_to_ol`가 수백 회 발생해 OL/INT 사이를 반복한다. 8T에서는 IOPS가 INT와 비슷해져 CP/LHP/PAS보다 낮고, 16T/20T에서는 device saturation 때문에 IOPS는 같지만 CPU는 낮다.
- 사용자가 지적한 대로 우선 봐야 할 것은 IOPS가 아니라 QD 변화다. 현재 디버깅용 stats는 시간순 trace가 아니라 run별 `last_qd`, `min_qd`, `max_qd`만 제공한다.
- 평균 QD 범위는 1T `1.00/1.00/1.00`, 2T `1.54/1.38/1.92`, 4T `2.78/2.18/3.26`, 8T `5.00/2.40/5.42`, 16T `11.16/1.96/12.12`, 20T `14.66/2.66/15.56`이다.
- 해석상 DPAS는 2T/4T에서 평균 QD가 1보다 높지만 `switch_param3=30` 기준 QD 3.0보다 대체로 낮아 PAS에 머문다. 8T 이상에서는 QD가 3.0을 넘어 OL에서 INT로 자주 넘어간다.
- 16T/20T에서 `min_qd`가 2~3 근처로 낮게 찍히는 것은 window 초반/전환 직후 QD가 낮은 구간이 섞였다는 뜻이다. 시간순 QD 변화를 보려면 현재 stats로는 부족하고 per-window QD trace를 추가해야 한다.

## 논문 4KRR Optane IOPS와 현재 migration 결과 비교

- 기준 수치는 repo root `REPRODUCE_EXCEL.xlsx`의 `4KRR Optane (nvme1n1)` sheet에서 읽었다.
- 현재 여러 CPU 결과는 논문/재현 스프레드시트 대비 대체로 60~68% 수준이다. 특히 saturation 구간은 논문 약 852~857 kIOPS, 현재 약 582 kIOPS다.
- 이 절대 격차는 `CP`와 `INT`에도 같이 나타난다. 따라서 현재 absolute IOPS 하락을 DPAS lock 추가만으로 설명하면 안 된다.
- CP 기준 정규화로 보면 현재 DPAS는 1T/16T/20T에서는 논문과 거의 같지만, 2T/4T/8T에서는 CP 대비 각각 약 8.4/7.0/11.7 percentage point 낮다.
- lock tradeoff 후보는 2T/4T의 DPAS-vs-PAS 소폭 손실에서 볼 수 있지만, 8T 손실은 QD가 높아 OL/INT 전환이 잦아진 정책 효과가 더 커 보인다.
