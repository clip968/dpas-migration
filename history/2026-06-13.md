### 공부한 개념

- `struct kiocb`: 커널이 한 개의 I/O 요청을 처리하는 동안 사용하는 "I/O ID 카드".
  - `ki_filp`: 사용자가 연 파일/디바이스 (`struct file`).
  - `ki_pos`: 현재 I/O offset.
  - `ki_flags`: `IOCB_HIPRI`, `IOCB_NOWAIT`, `IOCB_DIRECT` 등 사용자 요청 플래그.
  - `ki_complete`: AIO/io_uring 등 비동기 완료 콜백.
  - VFS read/write, direct I/O, libaio(`io_submit`), io_uring 모두 남부적으로 `kiocb`를 사용.
- `struct bio`: 블록 장치(NVMe, SATA 등)로 본내는 I/O 요청 패킷.
  - `bi_bdev`: 이 bio가 실제로 갈 block device.
  - `bi_opf`: 연산 종류(Read/Write) + 플래그(`REQ_POLLED`, `REQ_NOWAIT` 등).
  - `bi_iter`: 시작 sector, 길이 등 위치 정보.
  - `bi_io_vec`: 데이터가 들어 있는 물리적 페이지 배열.
  - `bi_end_io`: I/O 완료 시 호출될 콜백.
  - `bi_private`: `bi_end_io`에 넘길 추가 데이터.
- `kiocb` vs `bio`:
  - `kiocb`: VFS/파일 레이어의 "이번 I/O 요청 정보".
  - `bio`: 블록 레이어의 "디스크에 별내는 요청서".
  - 파일시스템 파일의 경우 `kiocb->ki_filp`는 파일을, `bio->bi_bdev`는 그 파일이 저장된 디스크를 가리킴.
  - raw block device의 경우 둘 다 같은 디스크를 가리키지만, `kiocb`는 open flags 등 파일 컨텍스트를 추가로 담음.
- `bio` vs `request`:
  - `bio`: 사용자/파일시스템이 만든 I/O 요청.
  - `request`: `blk-mq`가 `bio`를 변환/합쳐서 만든 스케줄링 단위.
  - 여러 `bio`가 하나의 `request`에 합쳐질 수 있음.
- DPAS 맥락에서의 `bio`:
  - `bio->bi_opf`의 `REQ_POLLED` 플래그로 polling/interrupt를 결정.
  - `bio_set_polled(bio, iocb)`는 `REQ_POLLED`를 세팅하고 polling 완료를 위해 `iocb`를 연결.
  - 5.18의 핵심 아이디어: mode가 INT면 `REQ_POLLED`를 끄고, CP/PAS/OL이면 켠다.

- warmup 이후 모든 mode의 절대 IOPS가 함께 올랐으므로, LHP 남부 효과로만 해석하면 안 된다. device/CPU/controller/실행 시점 환경 차이 가능성.

## 5.18 full DPAS 구현 구조 조사

사용자가 5.18에서 DPAS 로직이 실제로 어떻게 구현됐는지 코드 기준 설명을 요청했다.

확인한 5.18 구조:

- per-CPU `struct blk_switch`가 `enabled`, `mode`, mode별 counter, `switch_param1~7`, `qd/qd_sum`, `tf`를 가진다.
- sysfs `switch_enabled`는 모든 CPU의 `sc->enabled`를 켜고, `switch_param*`는 모든 CPU의 전이 threshold를 갱신한다.
- submit hook은 `kernel/fs/iomap/direct-io.c`와 `kernel/block/fops.c`에 분산되어 있다. `IOCB_HIPRI` I/O가 들어오면 `sc->mode`를 보고 `REQ_POLLED`를 유지하거나 지운다.
  - `_INT`이면 `IOCB_HIPRI`/`REQ_POLLED`를 지워 interrupt I/O로 강제한다.
  - `_CP`, `_PAS`, `_OL`이면 polled I/O로 제출하고 mode별 counter를 올린다.
- poll/PAS 경로(`kernel/block/blk-mq.c`)는 PAS sleep duration, `qd_sum`, `tf`를 갱신하고 `CP -> PAS`, `PAS -> OL`, `PAS -> CP`, `OL -> PAS`, `OL -> INT` 전이를 수행한다. `INT -> OL`은 submit hook에서 `int_cnt >= param7`일 때 일어난다.

주의:

- 5.18의 동작 아이디어는 중요하지만, 구현은 submit hook 중복, submit path 안 mode 전이, `N_POLL` 기반 약한 동기화 때문에 그대로 가져오면 위험하다.
