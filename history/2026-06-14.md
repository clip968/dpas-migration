# 2026-06-14

## 세션 시작 시 현재 위치 복구

이 파일은 2026-06-13 기록과 현재 `dpas-kernel` diff를 확인해서 새로 만든 일일 현황판이다.

### 현재 위치

- `dpas-kernel`에는 아직 커밋되지 않은 DPAS 코드 수정이 남아 있다.
- 수정 범위는 submit-path helper만이 아니라, `request_queue` 기반 DPAS mode 상태, `switch_enabled` sysfs knob, poll/PAS 경로의 mode 전환 일부까지 포함한다.
- 상위 repo에는 `dpas-kernel` 수정과 함께 `history/2026-06-13.md`, `history/DPAS-학습-노트.md`, `.kilo/`가 untracked 상태로 보인다.

### 이어받은 결정사항

- 5.18 DPAS 코드를 7.1로 포팅할 때는 핵심 아이디어만 가져오고 구조는 정리한다.
- mode 상태 저장은 5.18식 per-cpu `irq_poll_switch` 부활보다 `request_queue` 레벨로 통일하는 방향이 우선이다.
- submit path에서는 HIPRI I/O의 `REQ_POLLED` 여부만 결정하고, mode 전환은 poll/completion/PAS duration 업데이트 경로에서 처리한다.
- `switch_param1~7` 이름은 유지하되 의미 주석을 남기는 방향은 이미 반영됐다.

### 현재 코드상 중요한 위치

- `dpas-kernel/block/blk-core.c`: `blk_dpas_prepare_bio()` helper 추가.
- `dpas-kernel/block/fops.c`: raw block device async DIO submit 경로에서 helper 호출.
- `dpas-kernel/fs/iomap/direct-io.c`: filesystem direct I/O submit 경로에서 helper 호출.
- `dpas-kernel/include/linux/blkdev.h`: `request_queue`에 DPAS mode/counter/lock 필드 추가.
- `dpas-kernel/block/blk-sysfs.c`: `switch_enabled` sysfs show/store 및 상태 reset 추가.
- `dpas-kernel/block/blk-mq.c`: DPAS 상태 초기화, `tf`/`qd` 갱신, `blk_dpas_maybe_switch_mode()` 추가.

### 남은 리스크

- 현재 구현은 아직 빌드 검증되지 않았다. 오늘 시작 시점에는 `git diff --check`만 통과했다.
- mode 전환 로직이 원본 5.18 의도와 충분히 같은지, 그리고 7.1의 동시성/lock 구조에 맞는지는 추가 검토가 필요하다.
- `blk_dpas_prepare_bio()`가 `IOCB_HIPRI`를 지우는 경우 raw block DIO와 iomap DIO의 완료/대기 흐름이 모두 안전한지 확인해야 한다.
- CP mode에서 PAS sleep을 건너뛰는 현재 처리가 classic polling 의도와 맞는지 코드와 실측으로 확인해야 한다.
- `dpas_qd` 증감 위치가 실제 outstanding poll depth를 정확히 나타내는지 검토해야 한다.

### 다음에 볼 것

- 현재 diff를 기준으로 "submit helper까지만 첫 PR로 줄일지" 또는 "mode 전환까지 포함할지" 결정한다.
- `blk_dpas_prepare_bio()`가 INT mode에서 `IOCB_HIPRI`/`REQ_POLLED`를 끄는 동작을 각 submit 경로별로 다시 추적한다.
- `blk_dpas_maybe_switch_mode()`의 전이 조건을 5.18 원본 DPAS와 대조한다.
- 최소 검증으로 `dpas-kernel` 빌드 또는 관련 정적 selftest를 돌릴 범위를 정한다.

## 현재 DPAS 코드 검증

사용자가 현재 수정이 제대로 되었는지 검증을 요청했다.

확인 결과:

- `git diff --check`는 통과했다.
- `tools/testing/selftests/dpas/pas_queue_lifecycle_static.py`와 `request_queue_pas_stat_static.py`는 통과했다.
- `tools/testing/selftests/dpas/pas_sysfs_static.py`는 `switch_enabled sysfs must wait for its backing logic`로 실패했다.
  - 현재 코드는 `switch_enabled` backing logic을 추가한 상태라 기존 테스트 정책과 충돌한다.
- `tools/testing/selftests/dpas/no_q_dpas_static.py`는 `q->dpas` direct field 사용 때문에 실패했다.
  - 현재 설계가 `request_queue` direct field 방식이면 이 guard 테스트는 갱신이 필요하다.
- `/tmp/dpas-build` out-of-tree `x86_64_defconfig` 후 변경 object 컴파일을 시도했다.
  - 일반 빌드는 `include/linux/blkdev.h`의 `enum dpas_mode` 선언 위치 때문에 실패했다.
  - `KCFLAGS=-Wno-error`로 경고를 error에서 내리면 변경 object들은 컴파일됐다.

현재 판단:

- submit helper의 큰 방향은 5.18 full DPAS 의도와 맞다.
- 그러나 현재 코드는 그대로는 커널 기본 빌드를 통과하지 못하므로 "제대로 완료" 상태가 아니다.
- 우선 `enum dpas_mode`를 `struct request_queue` 밖으로 옮기거나 별도 타입 선언으로 정리해야 한다.
- 그 다음 현재 설계를 유지할지에 맞춰 `pas_sysfs_static.py`, `no_q_dpas_static.py`의 정책을 갱신할지 결정해야 한다.
