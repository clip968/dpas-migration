# 2026-06-27

## 세션 시작 시 현재 위치 복구

- 전날 핵심 주제는 sync direct I/O polled wait loop가 단일 CPU에서 `bio_poll()`을 양보 없이 반복할 수 있다는 분석이었다.
- 사용자가 선택한 후보는 `dpas-kernel/block/blk-mq.c`의 `__blk_hctx_poll()`에서 기존 `pas_max_no_lock` 값을 poll batch budget으로 연결하는 최소 수정이다.
- 현재 `dpas-kernel` checkout은 `d0fc0f34e single cpu livelock 문제 디버깅`이며, 루트 repo 기준 서브모듈 포인터가 `63a80d0b9..d0fc0f34e`로 이동한 상태다.

## 오늘 코드 리뷰 내용

- `__blk_hctx_poll()`는 `READ_ONCE(q->max_no_lock)`을 읽고, completion 없이 돈 `poll_count`가 기본 100 이상이면 `ret=0`으로 반환한다.
- 이 변경은 block layer 내부 busy polling 한 번의 길이를 제한한다는 점에서는 의도와 맞다.
- 다만 sync direct I/O 상위 루프인 `fs/iomap/direct-io.c`는 polled bio에서 `bio_poll()`이 0을 반환해도 바로 다시 호출하고, 그 사이에 `schedule()` 또는 `cond_resched()`를 호출하지 않는다.
- 따라서 이 수정만으로 단일 CPU progress 보장이 끝났다고 보기는 어렵다. 실제 hang 재현 조건에서 완주 여부, CPU 사용률, task stack 확인이 필요하다.
- duration clamp helper(`blk_mq_poll_pas_scale_duration()`)와 `pas_reset_stats` sysfs knob는 현재 변경 범위에 함께 포함돼 있으며, 빠른 정적 selftest는 통과했다.

## 남은 확인 사항

- 단일 CPU 재현 테스트에서 `__blk_hctx_poll()` 제한 이후 실제로 fio가 완주하는지 확인한다.
- 완주하지 않으면 `fs/iomap/direct-io.c` 상위 루프에 양보 지점을 둘지, block poll 경로에서 다른 반환 의미를 줄지 사용자가 결정해야 한다.
- `pas_max_no_lock`은 sysfs 최소값이 1이라 현재 코드의 `max_poll_count == 0` 비활성화 경로는 사용자가 sysfs로 선택할 수 없다. 디버그 knob로 유지할지, 0 비활성화를 허용할지 결정이 필요하다.

## 원큐 설치 스크립트 사용법

- 목적: 현재 `dpas-kernel` checkout을 host에 설치하고, 다음 부팅 1회만 새 DPAS 커널로 부팅되도록 예약한다.
- 실행 위치:

```bash
cd /home/urop1/dpas-migration
sudo ./scripts/dpas_install_host_kernel.sh
```

- 스크립트가 하는 일:
  - `scripts/dpas_build_host_kernel.sh`를 호출해 커널과 module을 빌드한다.
  - 빌드는 `SUDO_USER` 권한으로 실행해 build 산출물이 root 소유가 되지 않게 한다.
  - `/lib/modules/$KREL/`에 module을 설치한다.
  - `/boot/vmlinuz-$KREL`, `/boot/System.map-$KREL`, `/boot/config-$KREL`을 설치한다.
  - `depmod "$KREL"`과 `mkinitramfs -o /boot/initrd.img-$KREL "$KREL"`을 실행한다.
  - `xfs.ko`, `nf_tables.ko`의 `vermagic`가 `$KREL`과 맞는지 확인한다.
  - `/boot/grub/custom.cfg`를 백업하고 새 one-shot GRUB entry를 작성한다.
  - `grub-script-check`로 문법을 확인하고 `grub-reboot`로 다음 부팅 1회를 예약한다.
- 스크립트가 하지 않는 일:
  - `reboot`를 자동 실행하지 않는다.
  - 사용자가 상태를 확인한 뒤 직접 `sudo reboot`를 실행한다.
- 재부팅 전 확인할 것:

```bash
KREL=$(make -s -C dpas-kernel O=../build/dpas-kernel-host kernelrelease)
ls -lh /boot/vmlinuz-$KREL /boot/System.map-$KREL /boot/config-$KREL /boot/initrd.img-$KREL
modinfo -F vermagic /lib/modules/$KREL/kernel/fs/xfs/xfs.ko
modinfo -F vermagic /lib/modules/$KREL/kernel/net/netfilter/nf_tables.ko
grub-script-check /boot/grub/custom.cfg
strings /boot/grub/grubenv | grep next_entry
```

- 기대 상태:
  - `/boot` 파일 4개가 모두 존재한다.
  - 두 `vermagic` 출력이 `$KREL`로 시작한다.
  - `next_entry=DPAS host $KREL one-shot candidate`가 보인다.
  - 위 조건이 맞으면 `sudo reboot`로 다음 부팅 1회만 새 커널을 테스트한다.

## 새 DPAS host 커널 부팅 검증

- 새로 부팅된 커널은 `7.1.0-rc4-dpas-host-gd0fc0f34e9e6`이며, `/proc/cmdline`의 `BOOT_IMAGE`도 같은 커널 이미지를 가리킨다.
- 루트 파일시스템은 `/dev/sdc2`의 `ext4`로 마운트되어 있고, 현재 XFS 마운트는 없다.
- `xfs`는 현재 로드된 모듈에는 없지만, 현재 커널용 `/lib/modules/7.1.0-rc4-dpas-host-gd0fc0f34e9e6/kernel/fs/xfs/xfs.ko`가 존재하고 `modprobe -n -v xfs`도 해당 모듈 삽입 경로를 보여준다.
- UFW는 `systemctl is-active ufw` 기준 `active`, `systemctl is-enabled ufw` 기준 `enabled`이다.
- UFW가 사용하는 netfilter/nftables 계열 모듈(`nf_tables`, `nft_compat`, `nf_conntrack`, `x_tables`, `xt_conntrack`, `xt_LOG`, `xt_tcpudp` 등)은 `/proc/modules`에서 로드 상태로 확인됐다.
- `ufw status verbose`와 `dmesg`/커널 저널의 상세 오류 확인은 root 권한이 필요해서 현재 세션에서는 확인하지 못했다.

## CPU0 단일 CPU DPAS runner 결과 해석

- 실행 조건은 `sudo taskset -c 0 env DPAS_DEVICE_LIST=nvme1n1 DPAS_IO_MODE=INT,CP,LHP,PAS,DPAS DPAS_JOB_LIST=1,2,4,8 DPAS_RUNTIME=10 DPAS_REPEATS=1 ./run_host_dpas_optane.sh --clean`이다.
- fio 결과는 `err=0`이고, IOPS 표의 값은 `parse.py` 기준 kIOPS이다.
- `CP`는 1T~8T에서 약 116~117 kIOPS로 거의 고정됐다. CPU0 하나에서 busy-poll이 포화된 결과로 해석 가능하다.
- CPU 표는 fio의 `usr% + sys%` 값을 가져온 것이며, `--group_reporting`에서 thread 수가 늘면 100%, 50%, 25%, 12.5%처럼 나뉘어 보인다. 단일 CPU 전체 사용량 비교에는 thread 수를 곱해 봐야 한다.
- `DPAS`는 1T에서 `CP`와 비슷하고, 4T/8T에서 `INT`와 거의 비슷한 IOPS를 냈다. `dpas_switch_stats`도 1T는 `PAS <-> CP`, 4T/8T는 `OL <-> INT` 쪽 전환이 많아 이 결과와 방향이 맞다.
- `DPAS_REPEATS=1`이라 `std=0`은 안정성을 의미하지 않는다. 논문/최종 판단용이면 반복 3회 이상이 필요하다.

## CPU0 단일 CPU 1/2/4/8/16/20T 5회 결과

- 실행 조건은 같은 CPU0 고정 runner에 `DPAS_JOB_LIST=1,2,4,8,16,20`, `DPAS_REPEATS=5`를 준 것이다.
- 모든 fio 로그에서 명백한 `err>0`/fio failure 문자열은 발견되지 않았다.
- 평균 IOPS는 `CP`가 1T~20T에서 약 115~118 kIOPS로 고정되어 단일 CPU busy-poll 포화 패턴을 보인다.
- `PAS`도 2T 이상에서 약 119~120 kIOPS로 고정되어, 단일 CPU 다중 thread 조건에서는 확장되지 않는다.
- `INT`는 4T에서 약 197 kIOPS로 최고점을 찍고, 8T~20T에서는 약 177~185 kIOPS 범위로 완만히 내려간다.
- `DPAS`는 1T에서 `CP`와 비슷하고, 4T 이상에서는 `INT`에 가까운 성능을 낸다. 다만 2T는 약 119 kIOPS로 `INT` 136 kIOPS, `LHP` 156 kIOPS보다 낮다.
- `dpas_switch_stats` 기준 1T는 `PAS <-> CP`, 4T 이상은 `OL -> INT`/`INT -> OL` 전환이 반복된다. 2T는 `pas_to_ol=1` 이후 `ol_stay_between`이 약 15040회 누적되어 중간 상태에 머무르는 패턴이다.
- 5회 반복 표준편차는 대부분 작아 이번 성능 패턴은 재현성이 있다. 남은 판단 지점은 2T에서 DPAS가 왜 INT/LHP 쪽으로 가지 않는지 정책 조건을 볼지 여부다.
