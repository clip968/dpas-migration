# 2026-06-26

## 세션 시작 시 현재 위치 복구

- 최근 기록상 `dpas-kernel`의 핵심 미해결 지점은 PAS duration scaling overflow/div fault 원인 해석과, 이후 실제 host/VM 검증이다.
- 현재 이어서 보는 주제는 `fio pvsync2 + direct=1 + hipri` 조건에서 sync direct I/O polled wait loop가 단일 CPU에서 양보 없이 돌 수 있다는 분석이다.
- 관련 핵심 파일은 `dpas-kernel/fs/iomap/direct-io.c`의 `__iomap_dio_rw()` sync wait loop와 `dpas-kernel/block/blk-mq.c`의 `__blk_hctx_poll()` / `blk_mq_poll_bio()` 경로다.

## 남은 판단 지점

- 첨부 분석의 패치 후보는 커널 동작/정책 선택이므로 바로 수정하지 않는다.
- 먼저 실제 hang 상태에서 sysrq stack 또는 `perf top`으로 `bio_poll -> blk_mq_poll_bio -> __blk_hctx_poll -> nvme_poll` hot path가 맞는지 확인해야 한다.

## polling hang 수정 후보 비교

- `kernel/` 5.18 DPAS 코드와 `dpas-kernel/` 모두 `fs/iomap/direct-io.c`의 sync DIO wait loop는 polled bio에서 `bio_poll()`만 호출하고 `blk_io_schedule()`로 자지 않는 구조다. 따라서 `__iomap_dio_rw()`에 `TASK_RUNNING`/`cond_resched()`를 넣는 수정은 progress 보장에는 직접적이지만 5.18 원형과 가장 유사한 방향은 아니다.
- `kernel/` 5.18 DPAS의 classic poll loop도 completion, signal, error, one-shot, `need_resched()` 전까지 `cpu_relax()`로 반복한다. 현재 `dpas-kernel`의 `__blk_hctx_poll()` 구조와 이 점은 유사하다.
- `pas_max_no_lock`/`max_no_lock`은 5.18 호환 knob로 존재하고 기본값도 100이지만, 확인한 5.18 poll loop에서는 탈출 budget으로 쓰이지 않는다.
- 그러므로 기존 5.18과 더 가까운 최소 수정은 엄밀히 말하면 “둘 다 아님”이다. 굳이 고르면 generic `iomap`을 바꾸는 방법보다, block layer 안에서 기존 DPAS knob인 `max_no_lock`을 poll batch budget으로 연결하는 방법이 DPAS 이식 범위와 더 가깝다.

## 2번 적용 시 최소 코드 수정 방향

- 커널 코드는 아직 수정하지 않았다.
- 적용한다면 `dpas-kernel/block/blk-mq.c`의 `__blk_hctx_poll()`에서 `READ_ONCE(q->max_no_lock)`을 한 번 읽고, completion 없이 돈 `poll_count`가 그 값 이상이면 이번 `bio_poll()` 호출을 `ret=0`으로 끝내는 방식이 가장 좁다.
- `pas_max_no_lock` sysfs knob와 `q->max_no_lock = 100` 초기값은 이미 있으므로 sysfs/struct 추가는 필요 없다.
- 이 수정은 inner poll loop의 무한 spin을 끊지만, outer `__iomap_dio_rw()` wait loop의 명시적 yield는 추가하지 않으므로 단일 CPU hang 재현에서 별도 검증이 필요하다.
