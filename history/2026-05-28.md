# 2026-05-28 DPAS 현황판

## 현재 위치

Part 4는 예전의 PAS-only skeleton 중심 계획에서 다음 순서로 바뀐 상태다.

```text
sync DIO HIPRI path 복구
-> 5.18 LHP 포팅
-> PAS 포팅
-> cp/int/lhp/ehp/pas mode 비교 및 정리
```

현재 가장 앞에 있는 작업은 `dpas-kernel/fs/iomap/direct-io.c`에서 sync DIO + `IOCB_HIPRI`가 실제 `REQ_POLLED` bio와 `bio_poll()` wait loop로 이어지는지 확인하는 것이다.

## 전날에서 가져온 남은 리스크

- sync DIO HIPRI 복구는 object build 기준 코드 경로를 맞춘 단계다. 실제 runtime에서 `bio_poll() -> blk_mq_poll() -> nvme_poll()`까지 안정적으로 도는지는 아직 확인해야 한다.
- 최신 upstream이 sync DIO polling을 막은 이유가 있으므로 hang/timeout 가능성을 계속 의식해야 한다.
- `direct-io.c`에 들어간 설명 주석은 kernel style에 맞춰 짧은 block comment로 정리해야 한다.
- LHP/PAS sleep을 붙이기 전, sync DIO path와 io_uring path의 역할 차이를 분명히 해야 한다.
- `request_queue` 직접 필드 방식은 5.18 재현에는 좋지만, 장기적으로 cleanup 비용이 생긴다.

## 다음에 볼 것

1. `direct-io.c`의 sync DIO HIPRI 수정 내용을 다시 읽고, `dio->submit.poll_bio`와 `iocb->private`의 lifetime을 확인한다.
2. `bio_poll()`이 `bio->bi_cookie`를 어떻게 사용해 `blk_mq_poll()`로 내려가는지 다시 추적한다.
3. `blk_mq_poll()` 안에서 LHP/PAS hook을 어디에 둘 수 있는지 후보를 비교한다.
4. 5.18 `kernel/fs/iomap/direct-io.c`와 최신 `dpas-kernel/fs/iomap/direct-io.c`의 차이를 다시 비교한다.
5. LHP에서 hrtimer/bucket/switch mode가 어느 구조체에 저장되는지 5.18 코드를 읽는다.
6. PAS sleep-before-poll을 넣기 전에 "어떤 I/O에 딱 한 번 sleep할 것인가"를 정리한다.

## pvsync2 + HIPRI 런타임 확인

- 처음 실행한 `fio --ioengine=pvsync2 --direct=1 --hipri=1`은 `Operation not supported`로 실패했다.
- 원인은 DPAS 코드가 아니라 VM 루트 디스크 `vda`의 virtio-blk poll queue가 0개였기 때문이다. 당시 `io_poll=0`이고 `echo 1 > /sys/block/vda/queue/io_poll`도 `Invalid argument`로 거절됐다.
- libvirt 커널 부트 인자에 `virtio_blk.poll_queues=1`을 추가하고 도메인을 완전히 내렸다가 다시 시작했다.
- 재부팅 후 `virtio_blk virtio2: 3/0/1 default/read/poll queues`로 바뀌었고 `/sys/block/vda/queue/io_poll=1`을 확인했다.
- kprobe 이벤트로 `iomap_dio_rw`, `iomap_dio_bio_iter`, `blk_mq_submit_bio`, `bio_poll`, `blk_mq_poll`, `virtblk_poll` 진입 횟수를 비교했다.

```text
baseline: fio pvsync2 direct=1 hipri=0, runtime=5s
iomap_dio_rw       174421
iomap_dio_bio_iter 174421
blk_mq_submit_bio  174626
bio_poll                0
blk_mq_poll             0
virtblk_poll            0

hipri: fio pvsync2 direct=1 hipri=1, runtime=5s
iomap_dio_rw       208951
iomap_dio_bio_iter 208951
blk_mq_submit_bio  208952
bio_poll           208951
blk_mq_poll        208951
virtblk_poll     18045468
```

이 결과로 현재 VM에서는 sync direct I/O + `RWF_HIPRI`가 `iomap_dio_rw()`/`iomap_dio_bio_iter()` 경로를 탄 뒤 `bio_poll()` -> `blk_mq_poll()` -> virtio-blk poll backend까지 내려가는 것을 확인했다.

아직 `nvme_poll()`은 확인하지 않았다. 현재 VM의 실제 테스트 디스크가 virtio-blk라서 다음 단계에서 NVMe emulation 또는 passthrough 장치를 붙여 `blk_mq_poll()` -> `nvme_poll()`까지 별도 확인해야 한다.

## 런타임 테스트 진행 방식과 의미

이번 테스트의 목적은 "커널이 부팅된다"가 아니라, Part 4에서 복구한 sync direct I/O + HIPRI 경로가 실제 실행 시점에 polling path로 내려가는지 확인하는 것이었다.

진행 순서는 다음과 같았다.

```text
1. dpas-kernel 7.1.0-rc4 빌드
2. 빌드한 bzImage로 libvirt VM 직접 부팅
3. SSH 접속 가능한 상태 확인
4. fio 설치
5. pvsync2 + direct=1 + hipri=1 실행
6. 실패 원인이 DPAS 코드인지 장치 구성인지 분리
7. virtio-blk poll queue 활성화
8. kprobe로 함수 진입 횟수 비교
```

처음 `hipri=1` FIO가 `Operation not supported`로 실패했을 때 바로 코드 문제로 보지 않고, 먼저 블록 장치의 polling 지원 상태를 확인했다. 그 결과 `vda`의 `io_poll=0`, virtio-blk poll queue 0개가 원인이었다. 그래서 `virtio_blk.poll_queues=1`을 부트 인자에 추가하고 VM을 완전히 재시작한 뒤 다시 확인했다.

재실행 결과는 다음처럼 해석한다.

```text
hipri=0:
  direct I/O submit 경로는 돈다.
  bio_poll(), blk_mq_poll(), virtblk_poll()은 호출되지 않는다.

hipri=1:
  direct I/O submit 경로가 돈다.
  bio_poll(), blk_mq_poll(), virtblk_poll()이 호출된다.
```

이 테스트로 얻은 효과는 다음과 같다.

- sync DIO + `RWF_HIPRI`가 최신 `dpas-kernel`에서 완전히 죽은 경로가 아니라는 것을 runtime으로 확인했다.
- `iomap_dio_rw()` -> `iomap_dio_bio_iter()` -> `blk_mq_submit_bio()` -> `bio_poll()` -> `blk_mq_poll()`까지의 기본 연결을 확인했다.
- `hipri=0`과 `hipri=1`의 차이가 kprobe count로 분명히 갈리므로, 이후 LHP/PAS hook을 붙였을 때 "어느 경로에서 효과가 생겼는지" 비교할 기준선이 생겼다.
- 실패 시 먼저 장치 poll queue, `/sys/block/*/queue/io_poll`, 드라이버 부트 파라미터를 확인해야 한다는 디버깅 순서가 정리됐다.
- 지금 VM은 virtio-blk backend라 `virtblk_poll()`까지만 증명한다. 따라서 NVMe 계층의 최종 목표인 `nvme_poll()` 검증은 별도 NVMe 테스트 디스크가 필요하다는 경계도 명확해졌다.

## 다음 NVMe 테스트 방향

사용자 결정에 따라 다음은 A안으로 진행한다.

```text
A안: 기존 libvirt VM에 scratch NVMe 디스크 추가
```

이 방식의 이유는 다음과 같다.

- 현재 SSH, root filesystem, 빌드 커널, fio, tracefs 환경을 그대로 재사용할 수 있다.
- root disk는 계속 virtio-blk로 두고, 테스트용 scratch 디스크만 NVMe로 붙일 수 있어 기존 VM 복구 위험이 작다.
- 실제 확인 대상이 `/mnt/nvme-test` 같은 별도 mount point로 분리되므로, virtio-blk 결과와 NVMe 결과를 비교하기 쉽다.
- 물리 NVMe passthrough는 현재 호스트의 시스템 디스크와 충돌할 수 있으므로 우선 제외한다.

NVMe 테스트의 성공 기준은 다음으로 잡는다.

```text
fio pvsync2 direct=1 hipri=1 on NVMe scratch disk
-> iomap_dio_rw() 진입
-> bio_poll() 진입
-> blk_mq_poll() 진입
-> nvme_poll() 진입
```

반대로 `hipri=0`에서는 `bio_poll()`, `blk_mq_poll()`, `nvme_poll()` count가 0에 가까워야 한다. 이 비교가 되어야 이후 LHP/PAS sleep-before-poll hook을 NVMe polling 경로에 붙였을 때 성능 변화와 함수 경로 변화를 분리해서 해석할 수 있다.

## A안 NVMe scratch 디스크 테스트 결과

기존 libvirt VM에 QEMU emulated NVMe scratch 디스크를 추가하는 방식으로 진행했다. 호스트 QEMU 10.2.2는 `nvme`, `nvme-ns`, `nvme-subsys` 디바이스를 지원한다.

진행 방법은 다음과 같았다.

```text
1. libvirt default pool에 4G qcow2 생성
   dpas-nvme-scratch.qcow2

2. 기존 VM inactive XML에 NVMe 디스크 추가
   target dev='nvme0n1', bus='nvme', serial='dpasnvme001'

3. NVMe poll queue를 위해 커널 부트 인자 추가
   nvme.poll_queues=1

4. NVMe bus hotplug는 libvirt가 지원하지 않아 VM을 완전히 power-cycle

5. 게스트에서 /dev/nvme0n1 확인 후 ext4 생성 및 mount
   /mnt/nvme-test

6. /mnt/nvme-test/fio-nvme.bin 대상으로 FIO + kprobe 비교
```

현재 VM 디스크 상태는 다음과 같다.

```text
root disk    -> vda, virtio-blk, /dev/vda1 mounted on /
scratch disk -> nvme0n1, QEMU NVMe Ctrl, mounted on /mnt/nvme-test
```

부팅 후 NVMe poll queue 상태도 확인했다.

```text
cmdline includes nvme.poll_queues=1
nvme nvme0: 4/0/1 default/read/poll queues
/sys/module/nvme/parameters/poll_queues = 1
/sys/block/nvme0n1/queue/io_poll = 1
```

kprobe 비교 결과는 다음과 같다.

```text
baseline: fio pvsync2 direct=1 hipri=0 on /mnt/nvme-test, runtime=5s
iomap_dio_rw        93551
iomap_dio_bio_iter  93551
blk_mq_submit_bio   93717
bio_poll                0
blk_mq_poll             0
virtblk_poll            0
nvme_poll               0

hipri: fio pvsync2 direct=1 hipri=1 on /mnt/nvme-test, runtime=5s
iomap_dio_rw       117112
iomap_dio_bio_iter 117112
blk_mq_submit_bio  117114
bio_poll           117112
blk_mq_poll        117112
virtblk_poll            0
nvme_poll        12620267
```

이 결과로 `pvsync2 + direct=1 + hipri=1`이 NVMe scratch 디스크에서 `iomap_dio_rw()` -> `iomap_dio_bio_iter()` -> `blk_mq_submit_bio()` -> `bio_poll()` -> `blk_mq_poll()` -> `nvme_poll()`까지 실제로 내려가는 것을 확인했다.

이번 결과의 기대효과는 다음과 같다.

- Part 4의 sync DIO HIPRI 복구가 virtio-blk뿐 아니라 NVMe driver polling backend까지 연결되는지 runtime으로 확인했다.
- `hipri=0` 대비 `hipri=1`에서만 `bio_poll()`, `blk_mq_poll()`, `nvme_poll()`이 증가하므로, HIPRI 조건에 의해 poll path가 열리는 기준선이 생겼다.
- 테스트 대상 I/O가 NVMe scratch disk로 분리되어 `virtblk_poll()`이 0으로 유지됐다. 따라서 root disk virtio-blk 활동과 NVMe 테스트 I/O를 구분할 수 있다.
- 이후 LHP/PAS sleep-before-poll hook을 붙일 때 `blk_mq_poll()` 또는 `nvme_poll()` 주변에서 sleep/counter/latency 변화를 비교할 수 있는 최소 재현 환경이 생겼다.

한계도 있다. 이 디스크는 QEMU emulated NVMe이므로 실제 물리 NVMe latency나 hardware queue 특성을 대표하지는 않는다. 다만 kernel NVMe host driver의 `nvme_poll()` 경로 진입 여부를 확인하기에는 충분하다.

## PAS hook 1차 연결 상태

`nvme_poll()`까지 내려가는 runtime path를 확인한 뒤, 바로 전체 PAS 알고리즘을 붙이지 않고 최신 커널용 PAS 진입점을 먼저 만들었다.

핵심 판단은 다음과 같다.

```text
5.18 DPAS:
  blk_mq_poll(q, cookie, ...)
  -> cookie로 request를 복원
  -> request 기반으로 LHP/PAS sleep과 통계 처리

현재 dpas-kernel:
  bio->bi_cookie가 request tag가 아니라 hctx queue index 성격
  -> blk_mq_poll() 안에서 예전처럼 request를 안전하게 복원하기 어려움
  -> bio_poll()에서 bio를 blk-mq helper로 넘기는 새 연결부가 필요
```

현재 코드 변경은 `bio_poll()` 경로에 bio-aware wrapper를 추가한 것이다.

```text
bio_poll()
  -> blk_mq_poll_bio(q, bio, cookie, iob, flags)
      -> blk_mq_poll_pas_prepare(q, bio, flags)
      -> blk_hctx_poll(q, hctx, iob, flags)
          -> nvme_poll()
```

수정 파일과 의미는 다음과 같다.

- `dpas-kernel/block/blk-core.c`
  - mq queue일 때 `blk_mq_poll()` 대신 `blk_mq_poll_bio()`를 호출하도록 바꿨다.
  - 목적은 기존 `cookie`뿐 아니라 `bio`도 PAS helper까지 전달하는 것이다.
- `dpas-kernel/block/blk-mq.c`
  - `blk_mq_poll_bio()`를 추가했다.
  - `cookie`로 `hctx`를 찾은 뒤, PAS 준비 helper를 거쳐 기존처럼 `blk_hctx_poll()`로 내려간다.
  - `blk_mq_poll_pas_prepare()`는 아직 sleep 없이 `pas_enabled=1`일 때 `q->last_poll_count++`만 수행한다.
- `dpas-kernel/block/blk-mq.h`
  - `blk_mq_poll_bio()` prototype을 추가했다.

이 변경의 기대효과는 다음과 같다.

- `pas_enabled=0`이면 `blk_mq_poll_pas_prepare()`가 바로 return하므로 기존 polling backend 동작을 보존한다.
- `pas_enabled=1`이면 실제 `bio_poll()` 경로에서 PAS helper가 호출되는지 확인할 수 있다.
- 최신 커널에서 예전 5.18식 `cookie -> request` 복원에 매달리지 않고, 이후 bucket 계산과 sleep 판단에 필요한 `bio`를 helper에 전달할 수 있다.

검증 상태는 targeted object build 기준 통과다.

```text
git -C dpas-kernel diff --check
make -C dpas-kernel O=/tmp/dpas-build block/blk-core.o block/blk-mq.o
```

처음에는 `blk_mq_poll_bio(q, cookie, iob, flags)`처럼 `bio` 인자를 빠뜨려 빌드가 깨졌고, 이후 `blk_mq_poll_bio(q, bio, cookie, iob, flags)`로 수정해 `block/blk-core.o`와 `block/blk-mq.o` 컴파일을 통과했다.

아직 남은 정리는 다음과 같다.

- `blk_mq_poll_pas_prepare()`의 `if(!...)` spacing과 함수 선언 line wrap을 kernel style로 정리한다.
- 현재 `q->last_poll_count++`는 임시 checkpoint counter다. 멀티코어 fio에서 정확한 통계로 쓰기에는 race가 있으므로, 정식 PAS 통계는 이후 `pas_stat` 또는 별도 percpu/atomic counter로 다시 잡아야 한다.
- 전체 kernel build와 VM runtime 확인은 아직 이 변경 이후 다시 하지 않았다.

## 오늘 학습한 개념 정리

### DIO와 page cache

DIO는 Direct I/O이며, 일반 I/O와의 핵심 차이는 page cache를 우회한다는 점이다.

```text
Buffered I/O:
  application
  -> page cache
  -> filesystem/block layer
  -> device

Direct I/O:
  application
  -> filesystem direct I/O
  -> bio
  -> block layer
  -> device
```

PAS 실험에서 `direct=1`을 쓰는 이유는 page cache hit/writeback noise를 배제하기 위해서다. page cache가 끼면 read가 device까지 내려가지 않거나, write completion 시점과 실제 device writeback 시점이 분리될 수 있다. 그러면 `bio_poll()`, `blk_mq_poll()`, `nvme_poll()` 호출 여부와 polling sleep 효과를 제대로 해석하기 어렵다.

따라서 Part 4의 기준 경로는 다음처럼 잡는다.

```text
pvsync2 + direct=1 + hipri=1
-> iomap_dio_rw()
-> iomap_dio_bio_iter()
-> blk_mq_submit_bio()
-> bio_poll()
-> blk_mq_poll_bio()
-> nvme_poll()
```

### 일반 io_uring read/write와 NVMe uring passthrough

io_uring 전체가 request를 직접 내려보내는 것은 아니다.

일반 io_uring read/write는 `kiocb` 기반 file operation을 타고, direct I/O에서는 bio를 만든다.

```text
io_uring read/write
  -> file->f_op->read_iter/write_iter
  -> block/fops.c 또는 filesystem DIO
  -> iomap direct I/O
  -> bio
  -> submit_bio()
  -> blk_mq_submit_bio()
```

이 경우 polling은 다음처럼 bio 기반이다.

```text
io_uring iopoll
  -> file->f_op->iopoll()
  -> iocb_bio_iopoll()
  -> bio_poll()
  -> blk_mq_poll_bio()
```

반면 `NVME_URING_CMD_IO`는 일반 read/write가 아니라 NVMe passthrough uring command다. userspace가 NVMe command 필드에 가까운 구조를 채워 driver에 넘기고, NVMe driver가 직접 `struct request`를 만든다.

```text
NVMe uring passthrough
  -> nvme_uring_cmd_io()
  -> nvme_alloc_user_request()
  -> pdu->req = req
  -> blk_execute_rq_nowait()
```

polling도 `bio_poll()`이 아니라 request 기반이다.

```text
nvme_ns_chr_uring_cmd_iopoll()
  -> pdu->req
  -> blk_rq_poll(req, ...)
  -> blk_hctx_poll()
```

즉 `blk_rq_poll()`은 이미 `struct request *rq`를 들고 있는 caller를 위한 request-level polling 함수다. 현재 PAS 1차 hook은 `bio_poll()` 기반 DIO/HIPRI 경로를 대상으로 하므로 `blk_rq_poll()` 경로에는 걸리지 않는다. 나중에 NVMe passthrough uring command까지 PAS 대상으로 삼을 때 별도 request-aware hook 후보로 보면 된다.

## 다음 작업 갱신

가장 가까운 다음 단계는 PAS 알고리즘 전체 포팅이 아니라, 지금 만든 hook의 runtime 안전성을 확인하는 것이다.

```text
1. style cleanup
   -> blk_mq_poll_pas_prepare() spacing/comment/line wrap 정리

2. full kernel build
   -> targeted object build를 넘어서 bzImage까지 확인

3. VM runtime smoke
   -> pas_enabled=0에서 기존 bio_poll -> nvme_poll 경로 유지
   -> pas_enabled=1에서 blk_mq_poll_pas_prepare() 진입 확인

4. 그 다음 LHP/PAS sleep-before-poll 포팅
   -> no-sleep counter를 실제 hrtimer sleep으로 확장

```

## PAS 최소 hook VM runtime 검증

`blk_mq_poll_bio()` 최소 hook이 들어간 commit `c1ea80a5f73b` 기준으로 다시 빌드하고 VM에 반영했다.

빌드 검증은 다음을 통과했다.

```text
git -C dpas-kernel diff --check
python3 dpas-kernel/tools/testing/selftests/dpas/pas_sysfs_static.py
python3 dpas-kernel/tools/testing/selftests/dpas/request_queue_pas_stat_static.py
python3 dpas-kernel/tools/testing/selftests/dpas/pas_queue_lifecycle_static.py
python3 dpas-kernel/tools/testing/selftests/dpas/no_q_dpas_static.py
make -C dpas-kernel O=../build/dpas-kernel-vm olddefconfig
make -C dpas-kernel O=../build/dpas-kernel-vm block/blk-core.o block/blk-mq.o
make -C dpas-kernel O=../build/dpas-kernel-vm bzImage
```

새 VM 커널은 다음이다.

```text
uname -r -> 7.1.0-rc4-dpas-vm-gc1ea80a5f73b
cmdline  -> root=/dev/vda1 ... virtio_blk.poll_queues=1 nvme.poll_queues=1
```

주의할 점은 hook 이후 bio polling path가 기존 `blk_mq_poll()`을 직접 타지 않는다는 것이다. 새 경로는 다음처럼 봐야 한다.

```text
bio_poll()
-> blk_mq_poll_bio()
-> blk_hctx_poll()
-> nvme_poll()
```

`blk_mq_poll_pas_prepare()`는 컴파일러가 inline해서 별도 kallsyms 심볼로 보이지 않았다. 그래서 `objdump`로 `blk_mq_poll_bio()` 안의 `q->last_poll_count++` 위치를 확인했고, 해당 instruction인 `blk_mq_poll_bio+0x3a`에 kprobe를 걸어 `pas_increment`로 관측했다.

```text
blk_mq_poll_bio+0x3a:
  addl $0x1,0x48(%rdi)  # q->last_poll_count++
```

NVMe scratch disk `/mnt/nvme-test/fio-nvme.bin`에서 2초 FIO를 세 가지로 비교했다.

```text
case 1: pas_enabled=0, hipri=1
bio_poll          14421
blk_mq_poll           0
blk_mq_poll_bio   14421
blk_hctx_poll     14421
pas_increment         0
virtblk_poll          0
nvme_poll       8675471

case 2: pas_enabled=1, hipri=1
bio_poll          24971
blk_mq_poll           0
blk_mq_poll_bio   24971
blk_hctx_poll     24971
pas_increment     24971
virtblk_poll          0
nvme_poll       8017516

case 3: pas_enabled=1, hipri=0
bio_poll              0
blk_mq_poll           0
blk_mq_poll_bio       0
blk_hctx_poll         0
pas_increment         0
virtblk_poll          0
nvme_poll             0
```

해석은 다음과 같다.

- `pas_enabled=0`에서도 `bio_poll()` -> `blk_mq_poll_bio()` -> `blk_hctx_poll()` -> `nvme_poll()` 경로는 정상 유지됐다.
- `pas_enabled=0`에서는 `pas_increment=0`이라 checkpoint counter 분기가 실행되지 않았다.
- `pas_enabled=1`에서는 `pas_increment`가 `bio_poll()`/`blk_mq_poll_bio()`와 같은 횟수로 증가했다. 따라서 no-sleep PAS prepare hook이 실제 bio polling path에서 호출되고, enable 조건도 동작한다.
- `hipri=0`에서는 polling path 자체가 호출되지 않았다. 즉 hook은 일반 direct I/O path를 건드리지 않았다.
- `virtblk_poll=0`이라 테스트 I/O는 root disk가 아니라 NVMe scratch disk로 분리된 상태를 유지했다.

검증 후 상태도 확인했다.

```text
/sys/block/nvme0n1/queue/pas_enabled -> 0 으로 원복
/sys/kernel/tracing/kprobe_events    -> 0
/sys/kernel/tracing/dynamic_events   -> 0
dmesg err/warn tail                  -> 새 hook 관련 panic/WARN/lockup 없음
```

남은 정리는 style이다. `scripts/checkpatch.pl` 기준 다음 문제가 남았다.

```text
ERROR: space required before the open parenthesis '('
  if(!q->pas_enabled)
  if(!bio)

WARNING: line length of 105 exceeds 100 columns
  blk_mq_poll_bio() prototype

metadata warning:
  Missing commit description
  Missing Signed-off-by
```

다음 작업은 먼저 style cleanup을 한 뒤, `q->last_poll_count++` 임시 counter를 정식 PAS 통계 구조로 바꾸거나 sysfs/debug 출력으로 노출할지 결정하는 것이다. 지금 상태에서 runtime hook 연결 자체는 확인됐지만, `last_poll_count`는 race가 있는 임시 checkpoint이므로 성능/정확도 통계로 해석하면 안 된다.
